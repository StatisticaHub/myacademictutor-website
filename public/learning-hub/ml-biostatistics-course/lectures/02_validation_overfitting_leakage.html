Today’s big question

Why can a model look excellent during development but fail on new patients?

In medical machine learning, a model may appear strong for the wrong reasons.

This often happens because of:

  • overfitting;
  • poor validation;
  • data leakage;
  • using training performance as if it represents future performance.

A model is not useful just because it performs well on the data used to build it.

A useful model should perform well on new, unseen patients.

Intuition first

Imagine two students preparing for an exam.

One student memorises the exact answers from last year’s paper.
The other student understands the topic properly.

If the same questions appear again, the first student looks excellent.

But if new questions appear, the first student struggles.

A machine learning model can behave in the same way.

It may memorise the training data instead of learning patterns that generalise.

That is overfitting.

A model should learn useful patterns, not memorise the development dataset.

Training performance versus test performance

In prediction modelling, we usually separate model development from model evaluation.

Performance type Meaning
Training performance How well the model performs on data used to fit it
Test performance How well the model performs on new unseen data

Training performance is usually too optimistic because the model has already seen those patients.

Test performance is more realistic because it evaluates the model on patients not used during model fitting.

In medicine, the real question is not:

How well did the model fit the development data?

The real question is:

How well will the model perform on future patients?

What is overfitting?

Overfitting happens when a model learns patterns that are too specific to the training data.

These patterns may include:

  • random noise;
  • small sample quirks;
  • accidental associations;
  • measurement artefacts;
  • hospital-specific patterns;
  • time-period-specific patterns.

An overfitted model may look excellent during development but perform poorly when applied to new patients.

Medical example: hospital readmission prediction

Suppose we build a model to predict 30-day hospital readmission.

The model uses many predictors:

  • age;
  • diagnosis;
  • previous admissions;
  • medications;
  • lab results;
  • ward;
  • discharge notes;
  • follow-up appointments.

If the dataset is small and the model is too flexible, it may learn patterns that are specific to one hospital.

For example, it may learn that a particular ward has high readmission risk because of local workflow, not because the pattern generalises clinically.

When the model is used in another hospital, performance may collapse.

What is validation?

Validation means checking whether the model performs well beyond the data used to develop it.

There are different levels of validation.

Validation type Meaning
Apparent performance Performance on the same data used to fit the model
Internal validation Evaluation within the same dataset using splitting, cross-validation, or bootstrap
External validation Evaluation in a different dataset, hospital, time period, or population

Apparent performance is usually too optimistic.

Internal validation is better.

External validation is strongest because it asks whether the model works outside the original development setting.

Internal validation

Internal validation estimates how well the model might perform on new patients from a similar population.

Common methods include:

  • train/test split;
  • cross-validation;
  • bootstrap validation.

A train/test split is simple and easy to understand.

Cross-validation and bootstrap validation can be more efficient, especially when the dataset is not very large.

But all validation methods must be done carefully to avoid leakage.

External validation

External validation evaluates the model in a separate dataset.

This could mean:

  • another hospital;
  • another country;
  • another time period;
  • another clinical system;
  • another patient subgroup.

External validation is very important in medical prediction because patient populations and clinical practice can vary widely.

Clinical reality check

A model developed in one hospital may not work equally well in another hospital.

Differences in patients, measurement systems, clinical workflows, and disease prevalence can all affect model performance.

Enter Leakage Monster

Leakage Monster:
“I make models look better than they really are.”

Curious Learner:
“How?”

Leakage Monster:
“I let the model see information it would not have in real life.”

Prof Stat:
“That is called data leakage. It is one of the most dangerous mistakes in medical machine learning.”

Data leakage happens when information enters the model that would not be available at the proper prediction time point.

This can make performance look artificially strong.

Examples of data leakage

Data leakage can happen when:

  • test data are used during feature selection;
  • preprocessing is done before the train/test split;
  • future outcomes are included as predictors;
  • variables measured after diagnosis are used to predict diagnosis;
  • repeated measurements from the same patient are split incorrectly;
  • information from treatment decisions after the prediction time point enters the model;
  • outcome-related variables are accidentally included as predictors.

Medical example of leakage

Suppose we want to predict whether a patient will be admitted to ICU.

The prediction time point is:

At hospital admission

A bad predictor would be:

ICU discharge medication

Why?

Because ICU discharge medication is only known after ICU admission.

Using this variable would leak future information into the model.

The model may look excellent during evaluation, but it would be useless at the real prediction time point.

Prediction time point

The prediction time point answers:

When is the model supposed to make the prediction?

Examples:

At hospital admission
At diagnosis
Before treatment starts
At discharge
At first antenatal visit

Once the prediction time point is defined, only information available before or at that time should be used.

This is one of the simplest ways to reduce leakage.

Structured R workflows

Structured R workflows help reduce mistakes.

A good R workflow should:

  • split data before final model evaluation;
  • estimate preprocessing steps only on the training data;
  • apply the same preprocessing to the test data;
  • avoid using outcome information in predictor construction;
  • keep feature selection inside resampling when using cross-validation;
  • document the prediction time point clearly.

In later practical sessions, we will use R to build workflows that separate training and evaluation properly.

Common mistake 1

Common mistake: selecting predictors before splitting the data

Suppose you use the full dataset to select the strongest predictors.

Then you split the data into training and test sets.

This is a problem because the test set has already influenced the model-building process.

As a result, test performance may be too optimistic.

Common mistake 2

Common mistake: preprocessing before train/test split

Suppose you standardise variables using the mean and standard deviation from the full dataset.

Then you split into training and test sets.

This leaks information from the test data into the training process.

A safer workflow is:

  1. split the data;
  2. estimate preprocessing steps using only the training data;
  3. apply those steps to the test data.

Clinical reality check

Clinical reality check

If a model is overfitted or built with leakage, it may fail when used on real patients.

That could mean:

  • missed diagnoses;
  • unnecessary follow-up tests;
  • unsafe reassurance;
  • poor risk communication;
  • unfair performance across patient groups;
  • wasted clinical resources.

In medical machine learning, validation is not just a technical detail. It is part of patient safety.

Key takeaway

A prediction model should not be judged by training performance alone.

A reliable medical prediction model needs:

  • a clearly defined prediction time point;
  • proper separation of training and evaluation;
  • protection against data leakage;
  • assessment on unseen data;
  • careful interpretation of performance;
  • ideally, external validation.

Reflection question

You are building a model to predict 30-day hospital readmission at the time of discharge.

Which variables might be unsafe to include because they are only available after discharge or during the 30-day follow-up period?

Show suggested answer Unsafe variables may include information recorded after discharge, follow-up appointment outcomes, emergency department visits during the 30-day period, readmission-related notes, post-discharge medication changes, or any variable directly created using the future readmission outcome. Safe predictors should be available at or before discharge, such as age, diagnosis, previous admissions, comorbidities, length of stay, discharge medications, and discharge lab values if available at that time.
