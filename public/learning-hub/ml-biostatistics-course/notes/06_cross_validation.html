Cross-Validation in Medical Machine Learning

Learning objectives

By the end of this note, you should be able to:

  1. Explain why cross-validation is used in medical machine learning.
  2. Distinguish a single train/test split from k-fold cross-validation.
  3. Describe how k-fold cross-validation works.
  4. Explain why cross-validation is a form of internal validation.
  5. Recognise why preprocessing and feature selection must be done inside the cross-validation process.
  6. Understand why cross-validation does not replace external validation.
  7. Explain how cross-validation can be used for model comparison and tuning.
  8. Identify common mistakes that can make cross-validation results too optimistic.

1. Why this note matters

In previous notes, we discussed why validation is central to medical machine learning. A model should not be judged only by how well it performs on the data used to develop it. The important question is whether it can make useful predictions for new patients.

A simple train/test split is one way to estimate performance on unseen data. However, a single split can be unstable, especially when the dataset is small, the outcome is uncommon, or the patient population is heterogeneous. The result may depend too much on which patients happen to be placed in the training set and which patients happen to be placed in the test set.

Cross-validation helps address this problem. Instead of relying on one random split, it repeatedly splits the data into training and validation parts. This gives a more stable internal estimate of model performance.

The practical message of this note is:

Cross-validation helps estimate how well a model may perform on unseen patients from a similar source population, but it must be done carefully to avoid leakage.

2. The problem with one train/test split

A train/test split divides the dataset into two parts:

Training set: used to fit the model
Test set: used to evaluate the model

A common split is:

80% training
20% testing

This is simple and easy to understand. However, a single split may give a performance estimate that depends heavily on the random split.

For example, in a diabetes prediction study, one split may give:

AUC = 0.82

Another split may give:

AUC = 0.74

This does not necessarily mean the model has changed. It may mean that the test set changed. One test set may contain easier-to-predict patients, while another may contain more difficult cases.

This instability becomes more important when:

  • the sample size is small;
  • the number of outcome events is limited;
  • the outcome is imbalanced;
  • predictors are noisy;
  • the model is flexible;
  • there is strong patient heterogeneity.

3. What is cross-validation?

Cross-validation is an internal validation method.

It repeatedly divides the available data into training and validation parts. The model is fitted on the training part and evaluated on the validation part. This process is repeated several times, and the performance estimates are averaged.

Instead of asking:

How well does the model perform on one test split?

cross-validation asks:

How well does the model perform across several validation splits?

This usually gives a more stable estimate of internal performance.

4. k-fold cross-validation

The most common form is k-fold cross-validation.

In k-fold cross-validation, the dataset is divided into k approximately equal parts called folds.

For example, in 5-fold cross-validation:

  1. The data are split into 5 folds.
  2. The model is trained on 4 folds.
  3. The remaining fold is used for validation.
  4. This is repeated 5 times.
  5. Each fold is used once as the validation fold.
  6. The performance values are averaged.
Iteration 1: Train on folds 2,3,4,5; validate on fold 1
Iteration 2: Train on folds 1,3,4,5; validate on fold 2
Iteration 3: Train on folds 1,2,4,5; validate on fold 3
Iteration 4: Train on folds 1,2,3,5; validate on fold 4
Iteration 5: Train on folds 1,2,3,4; validate on fold 5

Each patient is used for validation once and training several times.

5. Why cross-validation is useful

Cross-validation is useful because it reduces dependence on one particular split.

It is commonly used for:

  • estimating internal model performance;
  • comparing different models;
  • tuning model parameters;
  • assessing model stability;
  • making better use of limited data.

For example, suppose we compare:

  • logistic regression;
  • ridge regression;
  • lasso regression;
  • random forest.

Cross-validation can estimate how each model performs across the same resampling structure.

This helps avoid choosing a model because it performed well on one lucky split.

6. Medical example: diabetes prediction

Suppose we want to predict diabetes status using:

  • glucose;
  • BMI;
  • age;
  • insulin;
  • blood pressure.

The outcome is:

Diabetes positive versus diabetes negative

If we use a single train/test split, the model may appear better or worse depending on the patients in the test set.

With 5-fold cross-validation, the model is evaluated across five validation folds. This gives a more stable view of how the model performs internally.

However, this still only tells us about performance within the same dataset. It does not prove that the model will work in another population.

7. Cross-validation is internal validation

Cross-validation is a form of internal validation.

Internal validation uses data from the same source as the development data. It estimates how the model might perform in similar patients from the same population.

It does not fully answer whether the model will work in:

  • another hospital;
  • another country;
  • another time period;
  • another laboratory system;
  • another ethnic group;
  • another healthcare setting.

That requires external validation.

A useful way to remember this is:

Cross-validation tests reuse within the same data source.
External validation tests transport to a new data source.

8. Cross-validation and data leakage

Cross-validation can still give misleading results if it is done incorrectly.

A common mistake is to perform preprocessing before cross-validation.

Examples include:

  • imputing missing values using the full dataset;
  • scaling variables using the full dataset;
  • selecting features using the full dataset;
  • performing PCA using the full dataset;
  • filtering biomarkers using the full dataset;
  • tuning model settings repeatedly using validation results.

If these steps are done before cross-validation, information from the validation folds leaks into the training process.

This can make performance look better than it really is.

9. The correct principle

The correct principle is:

Anything learned from data must be learned only from the training part of each fold.

This includes:

  • imputation;
  • scaling;
  • transformation;
  • feature selection;
  • model tuning;
  • model fitting.

The validation fold should behave like unseen data.

A better workflow is:

For each fold:
    1. Split into training fold and validation fold.
    2. Learn preprocessing only from the training fold.
    3. Apply preprocessing to the validation fold.
    4. Fit the model on the training fold.
    5. Evaluate on the validation fold.
Average performance across folds.

10. Feature selection inside cross-validation

Feature selection is a major source of leakage.

Suppose a dataset contains:

n = 250 patients
p = 20,000 gene expression features

An incorrect workflow is:

  1. Use the full dataset to select the 100 features most associated with the outcome.
  2. Run cross-validation using only those 100 features.
  3. Report cross-validated AUC.

This is biased because the validation folds helped choose the features.

A correct workflow is:

  1. Split into cross-validation folds.
  2. Within each training fold, select features.
  3. Fit the model using features selected from the training fold only.
  4. Evaluate on the validation fold.
  5. Repeat for each fold.

This is especially important in omics, imaging, and high-dimensional biomedical prediction.

11. Cross-validation for model tuning

Many machine learning models have tuning parameters.

Examples include:

  • penalty strength in ridge or lasso regression;
  • number of trees in a random forest;
  • maximum tree depth in boosting;
  • number of neighbours in k-nearest neighbours.

Cross-validation is often used to choose these tuning parameters.

However, tuning must be handled carefully. If the same validation results are repeatedly used to choose the best model and report final performance, the reported performance may become optimistic.

For strict model comparison, nested cross-validation may be needed.

12. Nested cross-validation

Nested cross-validation separates model tuning from model evaluation.

It has two layers:

Inner loop: choose tuning parameters
Outer loop: estimate model performance

This avoids using the same validation data both to tune the model and to estimate final performance.

Nested cross-validation is more computationally expensive, but it is useful when many models or tuning options are compared.

13. Repeated cross-validation

Cross-validation can be repeated several times.

For example:

10-fold cross-validation repeated 5 times

This means different random fold assignments are created repeatedly. Performance is averaged across all repeats.

Repeated cross-validation can reduce instability due to one particular fold assignment.

It is useful when the dataset is small or when model performance varies across folds.

14. Leave-one-out cross-validation

Leave-one-out cross-validation uses one observation as the validation set and all remaining observations as the training set.

If there are n patients, the model is fitted n times.

This uses nearly all data for training each time. However, it can be computationally expensive and may have high variance.

It is not always the best choice for medical prediction studies.

15. Cross-validation with repeated patient observations

Medical datasets often contain repeated observations.

For example:

  • multiple admissions per patient;
  • multiple scans per patient;
  • multiple visits per patient;
  • multiple samples per participant.

If rows are split randomly, data from the same patient may appear in both training and validation folds.

This causes leakage.

The correct approach is usually to split at the patient level, not the row level.

For example:

All records from Patient A should be in either training or validation, not both.

16. Cross-validation and class imbalance

Medical outcomes are often imbalanced.

For example, only 5% of patients may have the disease or event.

In such cases, folds should often be stratified so that each fold has a similar outcome distribution.

If one validation fold contains very few positive cases, performance estimates such as sensitivity or AUC may be unstable.

Stratified cross-validation helps reduce this problem.

17. Reporting cross-validation

When reporting cross-validation, describe:

  • number of folds;
  • whether cross-validation was repeated;
  • whether folds were stratified;
  • whether patient-level grouping was used;
  • what metrics were calculated;
  • how preprocessing was handled;
  • whether feature selection occurred inside folds;
  • how tuning was performed.

This is important for transparency and reproducibility.

18. Common mistakes

18.1 Feature selection before cross-validation

This leaks information from validation folds into model development.

18.2 Preprocessing before cross-validation

Imputation, scaling, or PCA using the full dataset can leak information.

18.3 Treating cross-validation as external validation

Cross-validation is internal validation only.

18.4 Reporting only the best fold

Performance should be summarised across all folds, not selected from the best fold.

18.5 Ignoring patient clustering

Repeated observations from the same patient must not be split across training and validation folds.

19. Summary

Cross-validation is a valuable internal validation method in medical machine learning. It is more stable than a single train/test split and can be used for model comparison and tuning.

However, cross-validation must be done carefully. Preprocessing, imputation, scaling, feature selection, and tuning should be included inside the resampling process. Otherwise, performance estimates may be too optimistic.

Cross-validation helps estimate performance in similar patients from the same source population, but it does not replace external validation.

Key terms

Term Meaning
Cross-validation Internal validation method using repeated training and validation splits
Fold One part of the data used in k-fold cross-validation
k-fold cross-validation Cross-validation where the data are divided into k folds
Validation fold Fold used to evaluate the model in one cross-validation iteration
Internal validation Validation within the same dataset or source population
External validation Validation in a separate dataset, setting, time period, or population
Data leakage Use of information that should not be available during model training or validation
Feature-selection leakage Leakage caused by selecting predictors before validation
Nested cross-validation Cross-validation with separate inner tuning and outer evaluation loops
Repeated cross-validation Cross-validation repeated with different fold assignments
Stratified cross-validation Cross-validation that preserves outcome proportions across folds
Patient-level splitting Splitting data so records from the same patient do not appear in both training and validation

Key takeaway

Cross-validation is useful because it gives a more stable internal estimate of model performance than one random split. However, it is only trustworthy when the full modelling workflow, including preprocessing and feature selection, is kept inside the validation process.
