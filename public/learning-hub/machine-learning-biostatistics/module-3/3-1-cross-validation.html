Today’s big question

Why do we use cross-validation in medical machine learning?

Cross-validation is a method for estimating how well a model may perform on unseen data. Instead of using only one training/test split, cross-validation repeatedly splits the data into training and validation parts.

Cross-validation gives a more stable internal estimate of model performance than relying on one random split.

Intuition first

Do not judge performance from one lucky question

Imagine testing a student using only one question.

The result may depend too much on that one question.

A better assessment uses several questions.

Cross-validation works similarly. It tests the model across several different splits of the data.

k-fold cross-validation

How k-fold cross-validation works

In k-fold cross-validation, the data are divided into k parts called folds.

For example, in 5-fold cross-validation:

  1. Split the data into 5 folds.
  2. Train the model on 4 folds.
  3. Validate it on the remaining fold.
  4. Repeat until each fold has been used for validation.
  5. Average the performance.
Iteration Validation fold Training folds
1Fold 1Folds 2, 3, 4, 5
2Fold 2Folds 1, 3, 4, 5
3Fold 3Folds 1, 2, 4, 5
4Fold 4Folds 1, 2, 3, 5
5Fold 5Folds 1, 2, 3, 4

Each patient is used for validation once and training several times.

Mini conversation

Cross-validation is internal validation

Curious Learner:
“So cross-validation gives me many test sets?”

Prof Stat:
“Not exactly. It gives repeated internal validation splits within the development data.”

Dr Clinic:
“So it is still not external validation?”

Prof Stat:
“Correct. Cross-validation is internal validation, not proof that the model works elsewhere.”

Cross-validation is useful when:

  • the dataset is not very large;
  • one train/test split may be unstable;
  • we want to compare models;
  • we want to tune model settings;
  • we want a better internal estimate of performance.

Clinical reality check

Cross-validation can improve internal validation, but it does not replace external validation. A model may perform well during cross-validation and still fail in a different hospital, country, time period, or patient group.

Medical example

Diabetes prediction

Suppose we are developing a diabetes prediction model.

A single train/test split may give:

AUC = 0.81

But another split may give:

AUC = 0.74

This variation happens because each split contains a different set of patients.

Cross-validation gives performance across multiple splits, which is more informative.

Common mistake

Feature selection before cross-validation causes leakage

Common mistake: doing feature selection before cross-validation

If you select predictors using the full dataset before cross-validation, information from the validation folds has already influenced model development.

This creates data leakage.

Feature selection, scaling, imputation and preprocessing should happen inside the cross-validation process.

Anything learned from data must be learned only from the training part of each fold.

Detailed notes

Note 6: Cross-Validation in Medical Machine Learning

Cross-validation helps estimate how well a model may perform on unseen patients from a similar source population, but it must be done carefully to avoid leakage.

1. Why this note matters

A simple train/test split is one way to estimate performance on unseen data. However, a single split can be unstable, especially when the dataset is small, the outcome is uncommon, or the patient population is heterogeneous. The result may depend too much on which patients happen to be placed in the training set and which patients happen to be placed in the test set.

Cross-validation helps address this problem. Instead of relying on one random split, it repeatedly splits the data into training and validation parts. This gives a more stable internal estimate of model performance.

2. The problem with one train/test split

A train/test split divides the dataset into two parts:

  • Training set: used to fit the model.
  • Test set: used to evaluate the model.

A common split is 80% training and 20% testing.

This is simple and easy to understand. However, a single split may give a performance estimate that depends heavily on the random split.

Split 1 AUC = 0.82 Split 2 AUC = 0.74

This instability becomes more important when the sample size is small, outcome events are limited, the outcome is imbalanced, predictors are noisy, the model is flexible, or there is strong patient heterogeneity.

3. What is cross-validation?

Cross-validation is an internal validation method. It repeatedly divides the available data into training and validation parts. The model is fitted on the training part and evaluated on the validation part. This process is repeated several times, and the performance estimates are averaged.

Instead of asking:

How well does the model perform on one test split?

cross-validation asks:

How well does the model perform across several validation splits?

4. k-fold cross-validation

In k-fold cross-validation, the dataset is divided into k approximately equal parts called folds. In 5-fold cross-validation, each fold is used once as the validation fold.

IterationTraining foldsValidation fold

5. Why cross-validation is useful

Cross-validation is commonly used for:

  • estimating internal model performance;
  • comparing different models;
  • tuning model parameters;
  • assessing model stability;
  • making better use of limited data.

6. Cross-validation is internal validation

Internal validation uses data from the same source as the development data. It estimates how the model might perform in similar patients from the same population.

It does not fully answer whether the model will work in another hospital, country, time period, laboratory system, ethnic group or healthcare setting.

Cross-validation tests reuse within the same data source. External validation tests transport to a new data source.

7. Cross-validation and data leakage

Cross-validation can still give misleading results if it is done incorrectly.

Common leakage sources include:

  • imputing missing values using the full dataset;
  • scaling variables using the full dataset;
  • selecting features using the full dataset;
  • performing PCA using the full dataset;
  • filtering biomarkers using the full dataset;
  • tuning model settings repeatedly using validation results without a separate evaluation design.

8. The correct principle

Anything learned from data must be learned only from the training part of each fold.

For each fold:
1. Split into training fold and validation fold.
2. Learn preprocessing only from the training fold.
3. Apply preprocessing to the validation fold.
4. Fit the model on the training fold.
5. Evaluate on the validation fold.
6. Average performance across folds.

9. Feature selection inside cross-validation

Feature selection is a major source of leakage.

n = 250 patients p = 20,000 gene expression features

An incorrect workflow is to select the 100 most associated features using the full dataset, then run cross-validation using only those features. This is biased because validation folds helped choose the features.

The correct workflow is to select features within each training fold only, fit the model in that training fold, and evaluate on the held-out validation fold.

10. Cross-validation for model tuning

Many machine learning models have tuning parameters, such as:

  • penalty strength in ridge or lasso regression;
  • number of trees in a random forest;
  • maximum tree depth in boosting;
  • number of neighbours in k-nearest neighbours.

Cross-validation is often used to choose tuning parameters, but if the same validation results are repeatedly used to choose the best model and report final performance, the reported performance may become optimistic.

11. Nested cross-validation

Nested cross-validation separates model tuning from model evaluation.

Inner loop

Tuning

Choose tuning parameters.

Outer loop

Evaluation

Estimate model performance.

12. Repeated and leave-one-out cross-validation

Cross-validation can be repeated several times, such as 10-fold cross-validation repeated 5 times. This can reduce instability due to one particular fold assignment.

Leave-one-out cross-validation uses one observation as the validation set and all remaining observations as training. It can be computationally expensive and is not always the best choice for medical prediction studies.

13. Repeated patient observations

Medical datasets often contain repeated observations, such as multiple admissions, scans, visits or samples per patient. If rows are split randomly, data from the same patient may appear in both training and validation folds. This causes leakage.

The correct approach is usually to split at the patient level, not the row level.

14. Class imbalance

Medical outcomes are often imbalanced. In such cases, folds should often be stratified so that each fold has a similar outcome distribution. If one validation fold contains very few positive cases, performance estimates such as sensitivity or AUC may be unstable.

15. Reporting cross-validation

When reporting cross-validation, describe:

  • number of folds;
  • whether cross-validation was repeated;
  • whether folds were stratified;
  • whether patient-level grouping was used;
  • what metrics were calculated;
  • how preprocessing was handled;
  • whether feature selection occurred inside folds;
  • how tuning was performed.

16. Common mistakes

  • Feature selection before cross-validation.
  • Preprocessing before cross-validation.
  • Treating cross-validation as external validation.
  • Reporting only the best fold.
  • Ignoring patient clustering.

17. Summary

Cross-validation is a valuable internal validation method in medical machine learning. It is more stable than a single train/test split and can be used for model comparison and tuning.

However, cross-validation must be done carefully. Preprocessing, imputation, scaling, feature selection, and tuning should be included inside the resampling process. Otherwise, performance estimates may be too optimistic.

Cross-validation helps estimate performance in similar patients from the same source population, but it does not replace external validation.

R code

R practical: 5-fold cross-validation

This script performs simple 5-fold cross-validation for a diabetes prediction model using logistic regression. It creates stratified folds, fits the model in each training fold, evaluates AUC in the validation fold, and summarises fold-to-fold variability.

Stratified folds

The script keeps diabetes positive and negative groups reasonably balanced across folds.

Fold AUC

Each fold produces its own AUC, showing that performance varies across validation splits.

Internal validation

The mean AUC summarises internal performance, but it is not external validation.

Download R script 

# Machine Learning in Biostatistics
# Script 06: Cross-Validation
#
# This script performs simple 5-fold cross-validation for a diabetes
# prediction model using logistic regression.

library(tidyverse)
library(mlbench)
library(pROC)

set.seed(123)

data("PimaIndiansDiabetes")

diabetes_data <- PimaIndiansDiabetes %>%
  mutate(
    diabetes = factor(diabetes, levels = c("neg", "pos"))
  )

# Create stratified folds manually.
# This keeps the positive and negative outcome groups reasonably balanced.

k <- 5

diabetes_data <- diabetes_data %>%
  group_by(diabetes) %>%
  mutate(fold = sample(rep(1:k, length.out = n()))) %>%
  ungroup()

cv_results <- map_dfr(1:k, function(current_fold) {
  train_data <- diabetes_data %>%
    filter(fold != current_fold) %>%
    select(-fold)

  validation_data <- diabetes_data %>%
    filter(fold == current_fold) %>%
    select(-fold)

  model <- glm(
    diabetes ~ pregnant + glucose + pressure + triceps +
      insulin + mass + pedigree + age,
    data = train_data,
    family = binomial
  )

  validation_data <- validation_data %>%
    mutate(
      predicted_risk = predict(model, newdata = validation_data, type = "response")
    )

  roc_object <- roc(
    response = validation_data$diabetes,
    predictor = validation_data$predicted_risk,
    levels = c("neg", "pos"),
    quiet = TRUE
  )

  tibble(
    fold = current_fold,
    auc = as.numeric(auc(roc_object)),
    n_validation = nrow(validation_data),
    events_validation = sum(validation_data$diabetes == "pos")
  )
})

cat("\nCross-validation results:\n")
print(cv_results)

cv_summary <- cv_results %>%
  summarise(
    mean_auc = mean(auc),
    sd_auc = sd(auc),
    min_auc = min(auc),
    max_auc = max(auc)
  )

cat("\nCross-validation summary:\n")
print(cv_summary)

plot(
  cv_results$fold,
  cv_results$auc,
  type = "b",
  ylim = c(0.5, 1),
  xlab = "Fold",
  ylab = "AUC",
  main = "AUC Across 5 Cross-Validation Folds"
)

abline(h = mean(cv_results$auc), lty = 2)

cat("\nLearning points:\n")
cat("1. Cross-validation evaluates the model across multiple splits.\n")
cat("2. It gives a more stable internal estimate than one split.\n")
cat("3. It is still internal validation, not external validation.\n")
cat("4. Preprocessing and feature selection should happen inside each fold.\n")

Exercises

Exercise 6: Cross-Validation

This exercise helps you understand why cross-validation is used in medical machine learning and how it differs from a single train/test split.

Question 1: Concept check

Explain in your own words why a single train/test split may give an unstable estimate of model performance.

Show suggested answer

A single split depends on which patients happen to enter the training and test sets. If the test set is easier or harder than average, the performance estimate may be too high or too low. This is especially problematic when the dataset is small, the event is rare, or patients are heterogeneous.

Question 2: Medical interpretation

A diabetes prediction model gives AUC = 0.82 in one split and AUC = 0.74 in another split. What does this suggest?

Show suggested answer

It suggests that the performance estimate is sensitive to the random split. Different test sets may contain different patient mixes, and one split may be lucky or unlucky. Cross-validation can provide a more stable internal estimate.

Question 3: True or false

Feature selection can be performed on the full dataset before cross-validation because cross-validation will still provide an unbiased performance estimate.

Show suggested answer

False. Selecting features using the full dataset lets validation folds influence model development. This is feature-selection leakage and can make cross-validated performance too optimistic.

Question 4: Applied reasoning

Suppose you are using 5-fold cross-validation. How many times is each patient used for validation?

Show suggested answer

Each patient is used for validation once. Each fold is held out once, and every patient belongs to exactly one validation fold.

Question 5: Short answer

Why does cross-validation not replace external validation?

Show suggested answer

Cross-validation reuses data from the same dataset or source population. It does not test whether the model works in another hospital, country, time period, laboratory system, or patient group. External validation is needed to assess transportability.

Question 6: Workflow problem

A researcher uses the full dataset to select the top 10 predictors, then performs 10-fold cross-validation using only those predictors. What is wrong?

Show suggested answer

The validation folds already influenced predictor selection. The reported cross-validated AUC is likely optimistic. Predictor selection should occur separately inside each training fold.

Question 7: Clinical reasoning

A model developed for hospital readmission has good cross-validated performance. Before using it in a new hospital, what additional evaluation would you want?

Show suggested answer

External validation in the new hospital or a comparable independent dataset. I would also check calibration, discrimination, threshold-specific performance, missing data patterns, patient mix, workflow differences, and whether predictors are measured in the same way.

Reflection

In a medical prediction study, what could go wrong if the cross-validation process is not carefully designed?

Show reflection answer

Poorly designed cross-validation can lead to overly optimistic results. Problems may include preprocessing before splitting, feature selection before cross-validation, repeated measurements from the same patient appearing in both training and validation folds, outcome information leaking into predictor construction, or tuning the model repeatedly until the best result is found.

In medical machine learning, this can make a model appear useful when it may fail on real patients.
