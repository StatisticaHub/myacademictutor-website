Lesson 20: XGBoost for Clinical Prediction

XGBoost is a fast, regularised and highly tunable implementation of gradient boosted trees. It can be powerful for clinical prediction, but high performance must be supported by validation, calibration, interpretability and clinical usefulness.

Today’s big question

How can XGBoost be used responsibly for clinical prediction?

XGBoost can fit flexible boosted tree models efficiently, but responsible medical use requires tuning, validation, calibration, interpretability checks and clinical usefulness assessment.

XGBoost architecture

What makes XGBoost different from basic boosting?

XGBoost extends gradient boosting with efficient computation, regularisation, subsampling, column sampling and early stopping. These controls make it powerful, but they also create more modelling decisions.

XGBoost works with numeric matrices

Unlike many formula-based R models, XGBoost usually expects predictors as a numeric matrix and outcome as a numeric label. For binary prediction, the outcome is often coded as 0 and 1.

Note 20: XGBoost for Clinical Prediction

XGBoost is powerful because it combines boosted trees with regularisation, subsampling and efficient computation. In medicine, this power must be matched with careful validation, calibration and interpretation.

1. Why this note matters

XGBoost is widely used in applied machine learning because it is fast, flexible and often highly predictive. It is especially popular for structured tabular data, which makes it relevant to many medical datasets.

Examples include hospital readmission prediction, ICU deterioration prediction, sepsis warning models, diabetes risk prediction, cancer recurrence prediction and treatment response prediction.

However, XGBoost should not be treated as automatically superior. In clinical prediction, a model is not useful simply because it is complex or popular. It must answer a clear prediction question and must be evaluated honestly.

2. What XGBoost means

XGBoost stands for extreme gradient boosting. It is an efficient implementation of gradient boosted decision trees.

Like gradient boosting, it builds trees sequentially. Each new tree improves the current model. XGBoost adds efficient computation, regularisation, subsampling, column sampling and other practical features.

3. Relationship to gradient boosting

XGBoost belongs to the boosting family. The core idea is still sequential improvement. The model starts with an initial prediction and adds trees to reduce prediction error.

The difference is that XGBoost is engineered to be faster, more flexible and more regularised than basic boosting implementations.

4. Why XGBoost is popular

XGBoost is popular because it can:

  • model non-linear relationships;
  • capture interactions;
  • handle many predictors;
  • use regularisation to control complexity;
  • work efficiently on tabular data;
  • provide feature importance summaries;
  • support early stopping.

5. Numeric matrix input

XGBoost usually works with numeric matrices. Each row represents a patient and each column represents a predictor. The outcome is stored separately as a numeric label, often 0 or 1 for binary classification.

In R, model.matrix() can be used to convert predictors into the required matrix format. Categorical predictors must be encoded appropriately.

6. Prediction-time availability

All predictors used by XGBoost must be available at the intended prediction time. If a predictor is measured after the clinical decision point, including it creates leakage.

This issue is not solved by using XGBoost. A powerful model can still be invalid if its predictors are not available when prediction is supposed to happen.

7. Objective function

The objective function tells XGBoost what task it is solving. For binary clinical prediction, binary:logistic is commonly used.

This produces predicted probabilities between 0 and 1. These probabilities should be interpreted as risks only after calibration has been assessed.

8. Evaluation metric

XGBoost can monitor metrics such as AUC, log loss or classification error during training.

AUC is useful for discrimination, but it does not assess calibration. Log loss and Brier score are more directly connected to probability prediction.

9. Learning rate

The learning rate, often called eta, controls how strongly each tree contributes.

A small learning rate makes learning gradual and usually requires more boosting rounds. A large learning rate learns quickly but may overfit or become unstable.

10. Maximum depth

Maximum depth controls how complex each tree can be. Deeper trees can learn more complex interactions.

In clinical prediction, very deep trees may learn unstable subgroup patterns. A moderate depth is often a reasonable starting point.

11. Number of boosting rounds

The number of rounds controls how many trees are added. Too few rounds may underfit. Too many rounds may overfit.

Early stopping helps choose a reasonable number of rounds by monitoring validation performance.

12. Subsampling

Subsampling uses only a fraction of patients for each tree. This can make the model less sensitive to the exact training data and can reduce overfitting.

Column sampling uses only a fraction of predictors for each tree. This can also reduce overfitting and make trees more diverse.

13. Regularisation

XGBoost includes regularisation parameters that penalise model complexity. These help prevent overly complex models from fitting noise.

Regularisation is useful but not a guarantee of generalisation. The model still requires validation.

14. Early stopping

Early stopping stops training when validation performance has not improved for a specified number of rounds.

It is a practical way to avoid adding unnecessary trees. However, the validation data used for early stopping must be separate from the final test set used for reporting performance.

15. XGBoost and overfitting

XGBoost can overfit despite regularisation. Overfitting is more likely with too many rounds, high learning rate, excessive depth, small node-size restrictions or repeated tuning on the same test set.

Overfitting may appear as strong internal performance but weaker external validation performance.

16. Feature importance

XGBoost can rank predictors by gain, cover or frequency. Gain measures how much a predictor improves the model when used in splits.

Feature importance should be interpreted as model importance, not causal importance. A feature may be important because it is a proxy, correlated with another variable, influenced by data collection practices or specific to a site.

17. Calibration

XGBoost may achieve good AUC but still produce poorly calibrated probabilities. This is critical because clinical decisions often depend on absolute risk.

Calibration should be assessed using plots, calibration slope, calibration intercept and Brier score where possible.

18. Threshold-based decisions

If a model is used to decide who receives further testing or treatment, a threshold probability is needed. For example, patients above 20% predicted risk may be referred for additional screening.

Threshold decisions depend on benefits and harms. A false negative may delay treatment, while a false positive may create unnecessary testing, anxiety or cost.

19. Decision curve analysis

Decision curve analysis can help evaluate whether an XGBoost model provides clinical net benefit across threshold probabilities.

This matters because high AUC does not automatically mean improved decisions.

20. Comparison with simpler models

XGBoost should be compared with simpler models such as logistic regression. If XGBoost only slightly improves AUC but is less interpretable and poorly calibrated, a simpler model may be preferable.

Model choice should consider performance, calibration, interpretability, implementation and clinical trust.

21. External validation

External validation is essential before strong clinical claims are made. XGBoost can adapt to local quirks in the development dataset, including measurement processes, case mix and coding patterns.

External validation evaluates whether performance holds in a different setting.

22. Reporting XGBoost models

A good report should include:

  • clinical prediction question;
  • outcome definition;
  • prediction time point;
  • candidate predictors;
  • matrix preparation and preprocessing;
  • training, validation and test split design;
  • objective function;
  • evaluation metric;
  • learning rate;
  • maximum depth;
  • number of rounds;
  • subsampling and column sampling;
  • regularisation settings;
  • early stopping procedure;
  • AUC, calibration and Brier score;
  • clinical usefulness assessment if relevant;
  • external validation if available;
  • limitations.

23. Common mistakes

  • Assuming XGBoost is automatically better because it is popular.
  • Tuning repeatedly on the final test set.
  • Reporting only AUC.
  • Ignoring calibration.
  • Interpreting feature importance as causality.
  • Using predictors unavailable at prediction time.
  • Ignoring external validation.
  • Overlooking implementation complexity in clinical workflows.

24. Summary

XGBoost is a powerful implementation of boosted trees. It can model complex patterns in medical data and often performs well in prediction tasks.

In clinical prediction, the key question is not whether XGBoost is powerful. The key question is whether the fitted model is valid, calibrated, interpretable enough, clinically useful and generalisable.

# Machine Learning in Biostatistics
# Script 20: XGBoost for Clinical Prediction
#
# This script fits an XGBoost model for diabetes prediction.
# It uses a training/test split, early stopping, AUC, Brier score,
# feature importance and a grouped calibration summary.
#
# Main ideas:
# 1. XGBoost is an efficient implementation of gradient boosted trees.
# 2. It can model non-linear relationships and interactions.
# 3. Regularisation and subsampling help control overfitting.
# 4. Early stopping helps choose the number of boosting rounds.
# 5. XGBoost still needs calibration and clinical validation.

library(tidyverse)
library(mlbench)
library(pROC)
library(xgboost)

set.seed(123)

data("PimaIndiansDiabetes")

diabetes_data <- PimaIndiansDiabetes %>%
  mutate(
    diabetes = factor(diabetes, levels = c("neg", "pos"))
  )

cat("\nDataset summary:\n")
cat("Number of patients:", nrow(diabetes_data), "\n")
cat("Outcome distribution:\n")
print(table(diabetes_data$diabetes))

train_index <- sample(
  seq_len(nrow(diabetes_data)),
  size = round(0.80 * nrow(diabetes_data))
)

train_data <- diabetes_data[train_index, ]
test_data <- diabetes_data[-train_index, ]

cat("\nTraining patients:", nrow(train_data), "\n")
cat("Test patients:", nrow(test_data), "\n")

x_train <- model.matrix(
  diabetes ~ pregnant + glucose + pressure + triceps +
    insulin + mass + pedigree + age,
  data = train_data
)[, -1]

x_test <- model.matrix(
  diabetes ~ pregnant + glucose + pressure + triceps +
    insulin + mass + pedigree + age,
  data = test_data
)[, -1]

y_train <- ifelse(train_data$diabetes == "pos", 1, 0)
y_test <- ifelse(test_data$diabetes == "pos", 1, 0)

dtrain <- xgb.DMatrix(
  data = x_train,
  label = y_train
)

dtest <- xgb.DMatrix(
  data = x_test,
  label = y_test
)

watchlist <- list(
  train = dtrain,
  test = dtest
)

xgb_model <- xgb.train(
  data = dtrain,
  objective = "binary:logistic",
  eval_metric = "auc",
  nrounds = 500,
  max_depth = 3,
  eta = 0.03,
  subsample = 0.80,
  colsample_bytree = 0.80,
  min_child_weight = 5,
  lambda = 1,
  alpha = 0,
  watchlist = watchlist,
  early_stopping_rounds = 25,
  verbose = 0
)

cat("\nBest iteration selected by early stopping:\n")
print(xgb_model$best_iteration)

cat("\nBest score reported by XGBoost:\n")
print(xgb_model$best_score)

xgb_risk <- predict(
  xgb_model,
  newdata = dtest
)

xgb_roc <- roc(
  response = test_data$diabetes,
  predictor = xgb_risk,
  levels = c("neg", "pos"),
  quiet = TRUE
)

xgb_auc <- as.numeric(auc(xgb_roc))
xgb_brier <- mean((xgb_risk - y_test)^2)

cat("\nXGBoost test AUC:\n")
print(xgb_auc)

cat("\nXGBoost test Brier score:\n")
print(xgb_brier)

plot(
  xgb_roc,
  main = "XGBoost ROC Curve",
  legacy.axes = TRUE
)

importance_table <- xgb.importance(
  feature_names = colnames(x_train),
  model = xgb_model
)

cat("\nXGBoost feature importance:\n")
print(importance_table)

xgb.plot.importance(
  importance_matrix = importance_table,
  main = "XGBoost Feature Importance"
)

xgb_predictions <- test_data %>%
  mutate(
    predicted_risk = xgb_risk,
    observed_numeric = y_test
  )

calibration_table <- xgb_predictions %>%
  mutate(risk_group = ntile(predicted_risk, 5)) %>%
  group_by(risk_group) %>%
  summarise(
    mean_predicted_risk = mean(predicted_risk),
    observed_risk = mean(observed_numeric),
    n = n(),
    .groups = "drop"
  )

cat("\nGrouped calibration summary for XGBoost:\n")
print(calibration_table)

plot(
  calibration_table$mean_predicted_risk,
  calibration_table$observed_risk,
  xlim = c(0, 1),
  ylim = c(0, 1),
  xlab = "Mean predicted risk",
  ylab = "Observed risk",
  main = "Grouped Calibration: XGBoost"
)

abline(0, 1, lty = 2)

logistic_model <- glm(
  diabetes ~ pregnant + glucose + pressure + triceps +
    insulin + mass + pedigree + age,
  data = train_data,
  family = binomial
)

logistic_risk <- predict(
  logistic_model,
  newdata = test_data,
  type = "response"
)

logistic_roc <- roc(
  response = test_data$diabetes,
  predictor = logistic_risk,
  levels = c("neg", "pos"),
  quiet = TRUE
)

model_comparison <- tibble(
  model = c("XGBoost", "Logistic regression"),
  test_auc = c(
    xgb_auc,
    as.numeric(auc(logistic_roc))
  ),
  test_brier = c(
    xgb_brier,
    mean((logistic_risk - y_test)^2)
  )
)

cat("\nModel comparison:\n")
print(model_comparison)

cat("\nLearning points:\n")
cat("1. XGBoost is a fast and regularised implementation of gradient boosted trees.\n")
cat("2. It uses tuning parameters such as learning rate, max depth, subsampling and regularisation.\n")
cat("3. Early stopping helps reduce overfitting by stopping when validation performance stops improving.\n")
cat("4. Feature importance describes predictive use, not causality.\n")
cat("5. AUC should be complemented by calibration, Brier score and clinical usefulness assessment.\n")
cat("6. External validation is needed before strong clinical claims are made.\n")
