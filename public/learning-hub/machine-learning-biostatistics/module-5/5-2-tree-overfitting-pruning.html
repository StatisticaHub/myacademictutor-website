Today’s big question

How do tree depth and pruning help control overfitting?

A decision tree can overfit if it grows too deep. Pruning removes unnecessary branches and can make the model more stable.

A decision tree becomes more complex every time it adds another split. Some splits capture real clinical structure, but others may capture random quirks of the development dataset.

Intuition first

A deeper tree is not always a better tree

Imagine a tree that creates a separate subgroup for almost every unusual patient in the training data. It may describe the development dataset extremely well, but those tiny rules may not apply to future patients.

The aim is not to make the largest possible tree. The aim is to find a tree that captures useful signal and generalises to new patients.

Tree depth

Depth controls how complex the rule can become

Tree depth is the number of splitting levels from the root node to the deepest terminal node. A depth of 1 gives one split. A depth of 2 allows another layer of splits. As depth increases, the model can represent more complex patient subgroups.

Moderate Balances simplicity and flexibility Often a reasonable starting point

Overfitting

High training performance can be a warning sign

A tree overfits when it learns patterns that are too specific to the training data. This usually appears as excellent training performance but weaker validation or test performance.

Overfitting is more likely when:

  • the sample size is small;
  • there are few outcome events;
  • many predictors are available;
  • the tree is allowed to grow deeply;
  • terminal nodes contain very few patients;
  • performance is judged only on training data.

Common mistake

Do not choose the deepest tree because it has the best training accuracy. Training accuracy is often optimistic because the tree was built using that same data.

Pruning

Pruning removes branches that are not worth keeping

Pruning is the process of cutting back a large tree to a smaller tree. The idea is to remove splits that add complexity without enough improvement in expected generalisation.

Grow a larger tree → estimate complexity-performance trade-off → remove weak branches → evaluate pruned tree

In the rpart framework, pruning is often controlled using a complexity parameter, usually called cp. A split is only kept if it improves the model enough.

Large cp

Smaller tree

Requires stronger improvement before a split is kept.

Small cp

Larger tree

Allows more splits and therefore more flexibility.

Pruning should be guided by validation or cross-validation rather than by appearance alone.

Medical example

Diabetes prediction: shallow, deep and pruned trees

Suppose we are predicting diabetes using glucose, BMI, age, blood pressure and related clinical variables.

Shallow tree

May use only glucose and BMI. It is easy to explain but may miss important patient subgroups.

Deep tree

May split repeatedly on many variables and create tiny groups. Training performance may look excellent.

Pruned tree

Removes weak branches to keep the most useful structure and reduce overfitting risk.

Clinical reality check

A tree with many highly specific rules may be hard to trust clinically. A simpler tree that performs similarly on test data may be more useful and safer to communicate.

Validation

Tree complexity should be judged on unseen data

To compare tree complexity, examine both training performance and test or validation performance. A large gap between training and test performance suggests overfitting.

1. Why this note matters

Decision trees are appealing because they produce visible rules. A clinician can often follow the path from the root node to a terminal node and understand how a patient was assigned to a risk group. This interpretability is valuable, especially in medical prediction.

However, interpretability does not guarantee reliability. A decision tree can be easy to draw and still be statistically weak. The main reason is that trees are highly adaptive. They search through many possible predictors and many possible cut-points. If we allow the tree to keep searching and splitting, it can find patterns that are specific to the development dataset rather than stable patterns that will apply to future patients.

This lesson therefore focuses on the central trade-off in decision trees: a tree must be flexible enough to capture useful clinical structure, but not so flexible that it memorises noise.

2. What tree depth means

Tree depth refers to the number of splitting levels in a tree. The root node is depth 0. After one split, the tree has depth 1. If the child nodes split again, the tree has depth 2, and so on.

A shallow tree has only a few splitting levels. It may be easy to explain, but it may be too simple to capture important clinical patterns. A deep tree has many splitting levels. It can represent complex combinations of predictors, but it may also create very small patient subgroups.

3. Why depth increases flexibility

Every additional split gives the model another opportunity to separate patients. For example, a diabetes tree may first split on glucose. Among high-glucose patients, it may split on BMI. Among high-glucose and high-BMI patients, it may split on age. This means later splits are conditional on earlier splits.

This conditional structure allows trees to model interactions naturally. The model can represent the idea that BMI may matter differently among high-glucose and low-glucose patients. This is useful, but the same flexibility can also produce overfitting.

4. Underfitting

Underfitting happens when a model is too simple to capture important signal. A tree with only one split may miss clinically meaningful patterns. For example, if a tree only splits on glucose, it may ignore useful information from BMI, age or diabetes pedigree.

Underfitting can produce poor training performance and poor validation performance. In this case, the model does not describe the development data well and also does not generalise well.

5. Overfitting

Overfitting happens when a model follows random noise or dataset-specific quirks rather than stable patterns. A deep tree can overfit by creating many small terminal nodes. Some leaves may contain only one or two patients. The estimated risk in these leaves is then based on very little information.

In medical prediction, overfitting is dangerous because it can make a model appear impressive during development but unreliable in future patients. A tree may have high training accuracy because it has effectively memorised the development data.

6. Training performance versus test performance

Training performance is measured on the same data used to build the tree. It is usually optimistic because the model was allowed to adapt to those patients. Test performance is measured on data not used to fit the tree, so it gives a more honest indication of generalisation.

A large difference between training and test performance is a warning sign. For example, if training AUC is 0.98 but test AUC is 0.68, the tree probably learned patterns that do not generalise.

7. Terminal node size

Terminal node size is the number of patients in a final leaf. Very small terminal nodes are unstable because their outcome proportion can change dramatically if only one patient is different.

For example, if a terminal node contains two patients and one has the outcome, the observed risk is 50%. If one additional patient is added, the risk estimate can change substantially. Larger terminal nodes usually produce more stable risk estimates.

8. Minimum split and minimum bucket rules

Tree algorithms often include rules such as a minimum number of patients required before a node can be split, and a minimum number of patients required in each terminal node. These rules prevent the tree from creating extremely small subgroups.

In clinical prediction, these rules are important because patient groups should be large enough to support reasonably stable risk estimates.

9. What pruning means

Pruning means cutting back a larger tree. Instead of keeping every split that improves training performance, pruning removes branches that do not provide enough expected benefit.

The idea is similar to choosing a simpler explanation when a more complicated explanation does not improve prediction in a meaningful way. A pruned tree may be easier to understand and may generalise better.

10. Pre-pruning and post-pruning

Pre-pruning controls the tree while it is being grown. Examples include setting a maximum depth, requiring a minimum number of patients in a node, or requiring a minimum improvement before a split is allowed.

Post-pruning first grows a larger tree and then cuts it back. The final tree is selected using a complexity-performance trade-off, often guided by cross-validation.

11. Complexity parameter

The complexity parameter controls how much improvement is required for a split to be kept. In rpart, this parameter is called cp.

A larger cp usually gives a smaller tree, because only stronger splits are retained. A smaller cp allows more splits, which can increase flexibility but also increase overfitting risk.

12. Complexity parameter table

The complexity parameter table shows how the tree changes as the pruning level changes. It usually includes the number of splits, relative error, cross-validated error and standard error.

A common approach is to choose the tree with the lowest cross-validated error, or a simpler tree whose error is within one standard error of the minimum. The second approach may produce a smaller and more stable tree.

13. The one-standard-error idea

The one-standard-error idea favours simplicity. If two trees have similar cross-validated performance, the simpler tree may be preferred because it is easier to interpret and may be more stable.

This is especially relevant in medicine, where overly complex rules can be difficult to explain, audit and implement.

14. Shallow tree versus deep tree versus pruned tree

A shallow tree may be too simple. It may miss important clinical subgroups and underfit the data.

A deep tree may be too complex. It may create many small groups and overfit the data.

A pruned tree aims to keep useful splits while removing weak branches. It is not automatically the best model, but it often provides a better balance between interpretability and generalisation.

15. Clinical example: diabetes prediction

In diabetes prediction, a shallow tree might split only on glucose. This is clinically sensible but may ignore additional risk information from BMI or age.

A deep tree might split on glucose, BMI, age, insulin, blood pressure, pedigree and several small subgroups. It may classify training patients well, but some of its branches may be based on very few patients.

A pruned tree may retain glucose and BMI splits while removing small unstable branches. This may be easier to explain and more likely to generalise.

16. Why pruning does not solve everything

Pruning reduces complexity, but it does not guarantee external validity. A pruned tree is still developed on one dataset. It may still be affected by sampling variation, measurement differences, case-mix differences or changes in clinical practice.

Therefore, after pruning, the model still needs validation, calibration assessment and careful clinical interpretation.

17. Calibration after pruning

Trees often produce step-like predicted probabilities because all patients in the same terminal node receive the same estimated risk. This can make calibration imperfect.

A pruned tree may be simpler and more stable, but its risk estimates may still be too high or too low in some risk groups. Calibration plots and Brier score are therefore important.

18. Choosing tree complexity without leakage

Tree complexity should not be chosen by repeatedly checking the final test set. If the test set is used to choose the depth or pruning level, it becomes part of model development.

A safer workflow is to tune tree complexity using training data or internal cross-validation, then evaluate once on the untouched test set.

19. Reporting a tree model

A clear report should describe:

  • candidate predictors;
  • outcome definition;
  • prediction time point;
  • tree-growing rules;
  • maximum depth or minimum node-size rules;
  • pruning method;
  • validation method;
  • discrimination;
  • calibration;
  • clinical interpretation;
  • limitations.

20. Common mistakes

  • Selecting the deepest tree because it has the highest training accuracy.
  • Reporting only training performance.
  • Ignoring calibration after pruning.
  • Assuming pruning removes all overfitting risk.
  • Choosing tree complexity by repeatedly checking the final test set.
  • Interpreting a tree split as causal evidence.
  • Preferring a complicated tree when a simpler tree performs similarly.

21. Summary

Tree depth increases flexibility. Too little depth can underfit. Too much depth can overfit. Pruning helps reduce complexity by removing weak branches, but the final model must still be validated honestly.

In medical prediction, the best tree is not necessarily the largest tree. It is the tree that gives a reasonable balance of performance, calibration, interpretability, clinical plausibility and generalisability.

R practical: shallow, deep and pruned trees

This script compares shallow, deep and pruned decision trees. It shows how overfitting appears when training AUC is high but test AUC is lower.

# Machine Learning in Biostatistics
# Script 17: Tree Overfitting, Tree Depth and Pruning
#
# This script shows why decision trees can overfit.
# It compares a shallow tree, a deep tree, and a pruned tree
# for diabetes prediction.
#
# Main ideas:
# 1. A shallow tree may be too simple.
# 2. A deep tree may fit training data too closely.
# 3. A pruned tree removes unnecessary branches.
# 4. Training performance and test performance should be compared.
# 5. Calibration should still be checked after choosing a tree.

library(tidyverse)
library(mlbench)
library(pROC)
library(rpart)
library(rpart.plot)

set.seed(123)

data("PimaIndiansDiabetes")

diabetes_data <- PimaIndiansDiabetes %>%
  mutate(
    diabetes = factor(diabetes, levels = c("neg", "pos"))
  )

cat("\nDataset summary:\n")
cat("Number of patients:", nrow(diabetes_data), "\n")
cat("Outcome distribution:\n")
print(table(diabetes_data$diabetes))

train_index <- sample(
  seq_len(nrow(diabetes_data)),
  size = round(0.80 * nrow(diabetes_data))
)

train_data <- diabetes_data[train_index, ]
test_data <- diabetes_data[-train_index, ]

cat("\nTraining patients:", nrow(train_data), "\n")
cat("Test patients:", nrow(test_data), "\n")

calculate_auc <- function(actual, risk) {
  roc_object <- roc(
    response = actual,
    predictor = risk,
    levels = c("neg", "pos"),
    quiet = TRUE
  )

  as.numeric(auc(roc_object))
}

calculate_brier <- function(actual, risk) {
  observed <- as.numeric(actual == "pos")
  mean((risk - observed)^2)
}

fit_tree <- function(cp, maxdepth, minsplit, minbucket) {
  rpart(
    diabetes ~ pregnant + glucose + pressure + triceps +
      insulin + mass + pedigree + age,
    data = train_data,
    method = "class",
    control = rpart.control(
      cp = cp,
      maxdepth = maxdepth,
      minsplit = minsplit,
      minbucket = minbucket
    )
  )
}

evaluate_tree <- function(model, model_name) {
  train_risk <- predict(
    model,
    newdata = train_data,
    type = "prob"
  )[, "pos"]

  test_risk <- predict(
    model,
    newdata = test_data,
    type = "prob"
  )[, "pos"]

  tibble(
    model = model_name,
    number_of_splits = sum(model$frame$var != "<leaf>"),
    terminal_nodes = sum(model$frame$var == "<leaf>"),
    train_auc = calculate_auc(train_data$diabetes, train_risk),
    test_auc = calculate_auc(test_data$diabetes, test_risk),
    train_brier = calculate_brier(train_data$diabetes, train_risk),
    test_brier = calculate_brier(test_data$diabetes, test_risk)
  )
}

shallow_tree <- fit_tree(
  cp = 0.02,
  maxdepth = 2,
  minsplit = 40,
  minbucket = 20
)

deep_tree <- fit_tree(
  cp = 0.0001,
  maxdepth = 20,
  minsplit = 2,
  minbucket = 1
)

cat("\nShallow tree complexity table:\n")
printcp(shallow_tree)

cat("\nDeep tree complexity table:\n")
printcp(deep_tree)

best_cp <- deep_tree$cptable[
  which.min(deep_tree$cptable[, "xerror"]),
  "CP"
]

pruned_tree <- prune(
  deep_tree,
  cp = best_cp
)

cat("\nBest complexity parameter selected from cross-validation error:\n")
print(best_cp)

cat("\nPruned tree complexity table:\n")
printcp(pruned_tree)

performance_comparison <- bind_rows(
  evaluate_tree(shallow_tree, "Shallow tree"),
  evaluate_tree(deep_tree, "Deep tree"),
  evaluate_tree(pruned_tree, "Pruned tree")
)

cat("\nTree performance comparison:\n")
print(performance_comparison)

performance_long <- performance_comparison %>%
  select(model, train_auc, test_auc) %>%
  pivot_longer(
    cols = c(train_auc, test_auc),
    names_to = "dataset",
    values_to = "auc"
  )

ggplot(performance_long, aes(x = model, y = auc, fill = dataset)) +
  geom_col(position = "dodge") +
  ylim(0, 1) +
  labs(
    title = "Training AUC versus Test AUC",
    x = "Tree model",
    y = "AUC"
  )

plotcp(
  deep_tree,
  main = "Complexity Parameter Plot for Deep Tree"
)

rpart.plot(
  shallow_tree,
  type = 2,
  extra = 104,
  fallen.leaves = TRUE,
  main = "Shallow Decision Tree"
)

rpart.plot(
  pruned_tree,
  type = 2,
  extra = 104,
  fallen.leaves = TRUE,
  main = "Pruned Decision Tree"
)

pruned_test_risk <- predict(
  pruned_tree,
  newdata = test_data,
  type = "prob"
)[, "pos"]

calibration_table <- test_data %>%
  mutate(predicted_risk = pruned_test_risk) %>%
  mutate(risk_group = ntile(predicted_risk, 5)) %>%
  group_by(risk_group) %>%
  summarise(
    mean_predicted_risk = mean(predicted_risk),
    observed_risk = mean(diabetes == "pos"),
    n = n(),
    .groups = "drop"
  )

cat("\nGrouped calibration summary for pruned tree:\n")
print(calibration_table)

plot(
  calibration_table$mean_predicted_risk,
  calibration_table$observed_risk,
  xlim = c(0, 1),
  ylim = c(0, 1),
  xlab = "Mean predicted risk",
  ylab = "Observed risk",
  main = "Grouped Calibration: Pruned Tree"
)

abline(0, 1, lty = 2)

cat("\nLearning points:\n")
cat("1. A shallow tree may underfit if it is too simple.\n")
cat("2. A deep tree may overfit if it follows noise in the training data.\n")
cat("3. Overfitting can appear as high training AUC but lower test AUC.\n")
cat("4. Pruning removes unnecessary branches and can improve generalisation.\n")
cat("5. A pruned tree still needs discrimination, calibration, and external validation assessment.\n")
