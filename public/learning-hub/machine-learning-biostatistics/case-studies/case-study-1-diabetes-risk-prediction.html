Case Study 1: Diabetes Risk Prediction

This case study applies supervised learning and logistic regression to a diabetes risk prediction problem. The aim is not only to fit a model, but to define the prediction question clearly, evaluate performance properly, and interpret the results in a clinically meaningful way.

Case study overview

What this case study is about

This case study applies supervised learning and logistic regression to a diabetes risk prediction problem. It shows the full modelling workflow from clinical question to model evaluation and interpretation.

The aim is to think like a medical statistician. A good prediction analysis is not finished when the model runs. We need to understand the outcome, check predictors, avoid leakage, evaluate the model honestly, interpret thresholds, examine calibration and explain limitations.

Outcome Diabetes status
Model Logistic regression
Task Binary classification
Dataset Pima Indians Diabetes

Core message

A model should not be judged by accuracy alone. In medical prediction, we need to consider discrimination, calibration, threshold-specific metrics, intended use and generalisability.

Clinical question

Can we predict whether a patient has diabetes?

The clinical question is:

Can we predict whether a patient has diabetes using routinely measured clinical characteristics?

This is a binary classification problem because the outcome has two categories: diabetes positive and diabetes negative.

In a real-world setting, a model like this could potentially support screening, risk stratification, or identifying patients who may need further testing. However, this case study is for teaching only and should not be interpreted as a deployable clinical model.

Why this case study matters

Diabetes risk prediction is a useful first case study because it brings together many issues that appear in real medical machine learning:

  • Is the outcome clearly defined?
  • Are the predictors clinically meaningful?
  • Are predictors available at the time of prediction?
  • Does the model perform well on unseen data?
  • Is sensitivity high enough for screening?
  • Is specificity high enough to avoid too many false positives?
  • Are predicted probabilities well calibrated?
  • Would the model generalise to another population?

Dataset

Pima Indians Diabetes dataset

This case study uses the Pima Indians Diabetes dataset, a commonly used teaching dataset for binary classification. The dataset includes diagnostic measurements and diabetes status.

Outcome

The outcome is diabetes status.

  • Positive class: diabetes positive
  • Negative class: diabetes negative

Defining the positive class clearly is essential because sensitivity, specificity, positive predictive value and negative predictive value all depend on which group is treated as the event.

Prediction time point

For this teaching example, the prediction time point can be understood as the time of clinical assessment. The model should only use information available at that moment.

Leakage reminder

If information measured after the prediction time point is used as a predictor, the model may look better than it truly is. This is data leakage.

Step 1

Outcome distribution

Before fitting a model, we examine the outcome distribution. This helps us understand whether the dataset is balanced or imbalanced. If one class is much more common, accuracy can become misleading.

Interpretation

The diabetes-negative group is larger than the diabetes-positive group. In the full dataset, there are approximately 500 diabetes-negative patients and 268 diabetes-positive patients. The outcome is moderately imbalanced.

This imbalance is not extreme, but it is enough that accuracy should not be used alone.

Steps 2–4

Train/test split and logistic regression model

The data are split into 80% training and 20% testing. The training set is used to fit the model. The test set is used to evaluate performance on unseen patients.

The logistic regression model estimates:

P(diabetes = positive | predictors)

Conceptually, the model is:

logit(P(diabetes positive)) = β0 + β1(pregnancies) + β2(glucose) + β3(blood pressure) + β4(triceps) + β5(insulin) + β6(BMI) + β7(pedigree) + β8(age)

Statistical interpretation

Glucose, BMI, pregnancies and diabetes pedigree function had stronger evidence of association with diabetes status than some other predictors in this fitted model. This should not be overinterpreted because it is one split of an educational dataset.

Steps 5–6

Predicted probabilities and classification results

Predicted probabilities are converted into predicted classes using a threshold. We begin with a threshold of 0.50:

Predicted probability ≥ 0.50 → diabetes positive
Predicted probability < 0.50 → diabetes negative

Clinical interpretation

At the 0.50 threshold, the model is better at identifying diabetes-negative patients than diabetes-positive patients. Specificity is high at 0.902, but sensitivity is lower at 0.577.

For a screening use case, this may be a problem because the model would miss a meaningful number of diabetes-positive patients.

Step 7

ROC curve and AUC

The ROC curve shows the trade-off between sensitivity and specificity across different thresholds. AUC measures discrimination: whether the model tends to assign higher predicted risks to patients with diabetes than to patients without diabetes.

AUC0.851
Brier score0.149

Interpretation

The model is reasonably good at ranking patients by diabetes risk. However, AUC does not tell us whether predicted probabilities are accurate, and it does not tell us which threshold should be used in practice.

Steps 8–9

Brier score and calibration

The Brier score measures the average squared difference between predicted probabilities and observed outcomes. Calibration assesses whether predicted probabilities agree with observed risks.

Calibration interpretation

A well-calibrated model should have points close to the diagonal line. In this case, the overall trend follows the diagonal reasonably, but some middle-risk groups move away from the line. This suggests calibration is somewhat unstable in the middle risk range.

Critical appraisal

What could go wrong?

1. Accuracy can be misleading

Accuracy is 0.792 at threshold 0.50, but sensitivity is only 0.577. A model can seem accurate while missing many positive cases.

2. Threshold may not match purpose

For screening, threshold 0.50 may be too conservative. Thresholds should reflect clinical consequences.

3. Generalisation is uncertain

Performance in one teaching dataset does not guarantee performance in another population or healthcare setting.

4. Calibration may be unstable

Some middle-risk groups deviate from the diagonal, so predicted probabilities should be interpreted cautiously.

5. Predictors may not be practical

A model is useful only if predictors are available, reliable and affordable at the prediction time point.

6. Dataset is educational

The Pima dataset is useful for teaching, but it should not be treated as a modern deployable clinical dataset.

Before real-world use

A diabetes risk model would require:

  • careful data quality assessment;
  • clinically justified predictor selection;
  • internal validation;
  • external validation;
  • calibration assessment;
  • threshold selection based on clinical consequences;
  • decision curve analysis;
  • fairness and subgroup performance checks;
  • transparent reporting.
