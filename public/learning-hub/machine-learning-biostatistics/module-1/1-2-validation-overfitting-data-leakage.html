Today’s big question

Why can a model look excellent during development but fail on new patients?

Why can a model look excellent during development but fail on new patients?

In medical machine learning, a model may appear strong for the wrong reasons.

This often happens because of:

  • overfitting;
  • poor validation;
  • data leakage;
  • using training performance as if it represents future performance.

A model is not useful just because it performs well on the data used to build it.

A useful model should perform well on new, unseen patients.

Intuition first

Memorising old answers is not the same as learning the topic

Imagine two students preparing for an exam.

One student memorises the exact answers from last year’s paper. The other student understands the topic properly.

If the same questions appear again, the first student looks excellent.

But if new questions appear, the first student struggles.

A machine learning model can behave in the same way.

It may memorise the training data instead of learning patterns that generalise.

That is overfitting.

A model should learn useful patterns, not memorise the development dataset.

Training versus test performance

Separate model development from model evaluation

In prediction modelling, we usually separate model development from model evaluation.

Performance type Meaning
Training performance How well the model performs on data used to fit it.
Test performance How well the model performs on new unseen data.

Training performance is usually too optimistic because the model has already seen those patients.

Test performance is more realistic because it evaluates the model on patients not used during model fitting.

In medicine, the real question is not:

How well did the model fit the development data?

The real question is:

How well will the model perform on future patients?

Overfitting

What is overfitting?

Overfitting happens when a model learns patterns that are too specific to the training data.

These patterns may include:

  • random noise;
  • small sample quirks;
  • accidental associations;
  • measurement artefacts;
  • hospital-specific patterns;
  • time-period-specific patterns.

An overfitted model may look excellent during development but perform poorly when applied to new patients.

Medical example

Hospital readmission prediction

Suppose we build a model to predict 30-day hospital readmission.

The model uses many predictors:

  • age;
  • diagnosis;
  • previous admissions;
  • medications;
  • lab results;
  • ward;
  • discharge notes;
  • follow-up appointments.

If the dataset is small and the model is too flexible, it may learn patterns that are specific to one hospital.

For example, it may learn that a particular ward has high readmission risk because of local workflow, not because the pattern generalises clinically.

When the model is used in another hospital, performance may collapse.

Validation

What is validation?

Validation means checking whether the model performs well beyond the data used to develop it.

There are different levels of validation.

Validation type Meaning
Apparent performance Performance on the same data used to fit the model.
Internal validation Evaluation within the same dataset using splitting, cross-validation, or bootstrap.
External validation Evaluation in a different dataset, hospital, time period, or population.

Apparent performance is usually too optimistic.

Internal validation is better.

External validation is strongest because it asks whether the model works outside the original development setting.

Internal and external validation

Validation should match the intended use

Internal validation

Internal validation estimates how well the model might perform on new patients from a similar population.

Common methods include:

  • train/test split;
  • cross-validation;
  • bootstrap validation.

A train/test split is simple and easy to understand.

Cross-validation and bootstrap validation can be more efficient, especially when the dataset is not very large.

But all validation methods must be done carefully to avoid leakage.

External validation

External validation evaluates the model in a separate dataset.

This could mean:

  • another hospital;
  • another country;
  • another time period;
  • another clinical system;
  • another patient subgroup.

External validation is very important in medical prediction because patient populations and clinical practice can vary widely.

Clinical reality check

A model developed in one hospital may not work equally well in another hospital.

Differences in patients, measurement systems, clinical workflows, and disease prevalence can all affect model performance.

Data leakage

Enter Leakage Monster

Leakage Monster:
“I make models look better than they really are.”

Curious Learner:
“How?”

Leakage Monster:
“I let the model see information it would not have in real life.”

Prof Stat:
“That is called data leakage. It is one of the most dangerous mistakes in medical machine learning.”

Data leakage happens when information enters the model that would not be available at the proper prediction time point.

This can make performance look artificially strong.

Examples of data leakage

Data leakage can happen when:

  • test data are used during feature selection;
  • preprocessing is done before the train/test split;
  • future outcomes are included as predictors;
  • variables measured after diagnosis are used to predict diagnosis;
  • repeated measurements from the same patient are split incorrectly;
  • information from treatment decisions after the prediction time point enters the model;
  • outcome-related variables are accidentally included as predictors.

Medical example

Leakage in ICU admission prediction

Suppose we want to predict whether a patient will be admitted to ICU.

The prediction time point is:

At hospital admission

A bad predictor would be:

ICU discharge medication

Why?

Because ICU discharge medication is only known after ICU admission.

Using this variable would leak future information into the model.

The model may look excellent during evaluation, but it would be useless at the real prediction time point.

Prediction time point

When is the model supposed to make the prediction?

The prediction time point answers:

When is the model supposed to make the prediction?

Examples:

  • At hospital admission
  • At diagnosis
  • Before treatment starts
  • At discharge
  • At first antenatal visit

Once the prediction time point is defined, only information available before or at that time should be used.

This is one of the simplest ways to reduce leakage.

Structured R workflows

Use workflow structure to reduce mistakes

Structured R workflows help reduce mistakes.

A good R workflow should:

  • split data before final model evaluation;
  • estimate preprocessing steps only on the training data;
  • apply the same preprocessing to the test data;
  • avoid using outcome information in predictor construction;
  • keep feature selection inside resampling when using cross-validation;
  • document the prediction time point clearly.

In later practical sessions, we will use R to build workflows that separate training and evaluation properly.

Common mistakes

Two mistakes that make performance too optimistic

Common mistake 1: selecting predictors before splitting the data

Suppose you use the full dataset to select the strongest predictors.

Then you split the data into training and test sets.

This is a problem because the test set has already influenced the model-building process.

As a result, test performance may be too optimistic.

Common mistake 2: preprocessing before train/test split

Suppose you standardise variables using the mean and standard deviation from the full dataset.

Then you split into training and test sets.

This leaks information from the test data into the training process.

A safer workflow is:

  • split the data;
  • estimate preprocessing steps using only the training data;
  • apply those steps to the test data.

If a model is overfitted or built with leakage, it may fail when used on real patients.

That could mean:

  • missed diagnoses;
  • unnecessary follow-up tests;
  • unsafe reassurance;
  • poor risk communication;
  • unfair performance across patient groups;

In medical machine learning, validation is not just a technical detail. It is part of patient safety.

Key takeaway

Do not judge a model by training performance alone

A prediction model should not be judged by training performance alone.

A reliable medical prediction model needs:

  • a clearly defined prediction time point;
  • proper separation of training and evaluation;
  • protection against data leakage;
  • assessment on unseen data;
  • careful interpretation of performance;
  • ideally, external validation.

Detailed notes

Note 2: Validation, Overfitting and Data Leakage

This note explains why validation is central to medical machine learning, how overfitting creates false optimism, and how data leakage can make a model look clinically useful when it would fail in practice.

Learning objectives

By the end of this note, you should be able to:

Explain why validation is central to machine learning in biostatistics.

Distinguish training performance, apparent performance, internal validation performance, and external validation performance.

Describe overfitting in medical prediction modelling and explain why it is common in clinical and biomedical datasets.

Explain the bias-variance trade-off using medical examples.

Distinguish internal validation from external validation and understand when each is useful.

Recognise common forms of data leakage in medical machine learning.

Explain why leakage can make a model appear clinically useful when it would fail in practice.

Understand why validation, calibration, reporting, and risk-of-bias assessment should be considered together in medical prediction modelling.

1. Why this note matters

In the first note, we introduced the distinction between prediction and inference. That distinction is essential because it changes how we judge a model. If the aim is inference, we may care mainly about an estimated association, a confidence interval, a p-value, or a causal interpretation. If the aim is prediction, we care mainly about how well the model performs for individuals who were not used to build the model.

This second note focuses on one of the most important ideas in medical machine learning: a model is only useful if it generalises to new patients. A prediction model can look impressive in the dataset where it was developed, but fail when applied to another hospital, another time period, another ethnic group, another laboratory platform, or another healthcare system. This gap between development performance and real-world performance is one of the major reasons why many medical machine learning models do not translate into clinical practice.

In biostatistics, validation is not a cosmetic final step. It is part of the scientific argument that a model may be reliable. If a study reports excellent model performance but does not describe how the model was validated, whether preprocessing was done correctly, whether predictors were available at the intended prediction time, or whether calibration was checked, then the reported performance may not be trustworthy. This is why modern clinical prediction model guidance, including TRIPOD+AI and PROBAST+AI, emphasises transparent reporting, validation, risk of bias, and applicability (Collins et al., 2024; Moons et al., 2025).

The practical message of this note is simple:

In medical machine learning, the goal is not to fit the best-looking model on the current dataset. The goal is to build and evaluate a model that can make reliable predictions for future patients in the intended clinical setting.

2. From model fitting to model validation

When we fit a model, we use data to estimate a relationship between predictors and an outcome. In supervised learning, the outcome is known during training. For example, we may have a dataset of patients with predictors such as age, sex, blood pressure, laboratory values, tumour stage, or biomarkers, and an outcome such as mortality, disease status, readmission, or treatment response.

A model may learn patterns from this dataset. Some of these patterns may be clinically meaningful. Others may be accidental. For example, a hospital dataset may contain coding habits that are specific to that hospital. A laboratory dataset may contain batch effects that are related to when samples were processed. A small biomedical dataset may contain chance correlations between thousands of biomarkers and the outcome. A model that is flexible enough may learn all of these patterns, even if they will not hold in future patients.

This is why model fitting and model validation are different tasks. Model fitting asks:

  • What relationship can the model learn from the development data?

Model validation asks:

  • How well does the fitted model perform when applied to data not used to develop it?

In statistical learning, the central issue is often not whether a model can fit the training data, but whether it can predict well on new observations (James et al., 2023). In medical statistics, this point is even more important because clinical decisions may depend on predicted risks. A model that appears accurate but is not valid can misclassify patients, overestimate or underestimate risk, and potentially support harmful decisions.

3. Training performance, apparent performance, and test performance

A major source of confusion in machine learning is the difference between performance estimated on the data used to fit the model and performance estimated on independent data.

3.1 Training performance

Training performance is the performance measured on the same data used to fit the model. If a logistic regression, random forest, or XGBoost model is trained on 1,000 patients and then evaluated on those same 1,000 patients, the resulting performance is training performance.

Training performance is usually optimistic because the model has already seen the data. This optimism may be small for simple models in large datasets, but it can be severe for flexible models, small datasets, rare outcomes, or high-dimensional predictors.

3.2 Apparent performance

In clinical prediction modelling, performance estimated directly in the development dataset is often called apparent performance. It is the performance the model appears to have before properly correcting for optimism.

Apparent performance can be misleading because it includes the benefit of all modelling choices made using the data. These choices may include variable selection, transformation choices, interaction exploration, hyperparameter tuning, threshold selection, and model comparison. Even if the final model is evaluated only once, the modelling process may have indirectly used the data many times.

3.3 Test performance

Test performance is estimated using data not used to fit the model. A test set is intended to mimic future patients. If the model performs well on an untouched test set, this provides stronger evidence that the model has learned patterns that generalise beyond the training data.

However, not all test sets are equally informative. A random split from the same dataset gives information about performance in similar data from the same source, but it may not tell us whether the model will work in another hospital or cohort. External validation is usually more informative for clinical generalisability.

4. A simple example: why training performance can mislead

Imagine a team develops a model to predict 30-day mortality after hospital admission. They use 500 patients from one hospital and try many machine learning algorithms. They report the following result:

Training AUC = 0.97

At first glance, this looks excellent. But we should immediately ask several questions:

Was the model evaluated on the same patients used to train it?

Were predictors selected before validation?

Were missing values imputed using the whole dataset?

Were any predictors measured after admission?

Was the model tested in another dataset?

Was calibration assessed?

Was the target population clearly defined?

If the answer is that the model was evaluated only in the training data, then the AUC of 0.97 tells us very little about how the model will perform for future patients. When the same model is tested on a later set of patients, the result might look like this:

Training AUC = 0.97 Internal test AUC = 0.76 External validation AUC = 0.64

The training AUC suggested near-perfect discrimination, but external validation suggests only modest performance. This pattern is not rare in medical machine learning. It may reflect overfitting, leakage, differences between populations, differences in measurement, poor calibration, or a combination of these issues.

5. Overfitting

Overfitting occurs when a model learns patterns that are too specific to the training dataset. Instead of learning general relationships that apply to future patients, the model learns noise, accidental correlations, artefacts, or dataset-specific structures.

A model that overfits will often show:

High training performance

Lower validation performance

Poor external performance

In medical statistics, overfitting is not just a technical inconvenience. It can lead to overconfident claims about clinical usefulness. If an overfitted model is reported as highly accurate, clinicians and researchers may believe that it can support decision-making, when in reality it may fail in practice.

5.1 Why overfitting happens

Overfitting becomes more likely when:

the sample size is small;

the number of outcome events is low;

the number of candidate predictors is large;

many models are tried and only the best result is reported;

hyperparameters are tuned without proper validation;

feature selection is performed using the full dataset;

flexible models are used without enough data;

the dataset contains batch effects, repeated observations, or hidden clusters;

outcome definitions or predictor definitions are inconsistent.

These conditions are common in medical datasets. Many studies have limited sample sizes because collecting high-quality clinical data is difficult. Outcomes may be rare. Biomedical data may contain tens of thousands or hundreds of thousands of predictors. Data may come from one hospital, one assay platform, or one time period. All of these features increase the risk that the model will learn unstable patterns.

6. Overfitting in clinical prediction: an example

Suppose a researcher wants to predict 1-year mortality in cancer patients using baseline clinical variables and biomarkers.

The dataset has:

n = 180 patients Outcome events = 35 deaths Candidate predictors = 120 variables

The researcher fits several models: logistic regression, lasso, random forest, gradient boosting, and support vector machine. They try many tuning settings and choose the model with the highest AUC. The best model has:

Training AUC = 0.99

This result should be treated with caution. With only 35 outcome events and 120 candidate predictors, a flexible model can easily learn patterns that are specific to the dataset. If the model was not validated properly, the AUC is likely to be optimistic. If the researcher selected predictors before splitting the data, the result may be affected by leakage. If the model was tuned repeatedly on the test set, the test performance may also be optimistic.

A more responsible analysis would define the prediction question clearly, restrict predictors to those available at the intended prediction time, use internal validation or resampling correctly, assess calibration, and ideally test the final model in an external cohort. This approach is aligned with the clinical prediction modelling perspective described by Steyerberg (2019) and with modern reporting and risk-of-bias guidance (Collins et al., 2024; Moons et al., 2025).

7. Bias-variance trade-off

The bias-variance trade-off is a central concept in statistical learning. It helps explain why both overly simple and overly complex models can perform poorly.

A useful way to think about expected prediction error is:

Prediction error = bias + variance + irreducible error

This is a conceptual decomposition rather than something we usually calculate directly in routine medical modelling. It is still useful because it explains why model complexity must be controlled.

7.1 Bias

Bias refers to error from overly simplistic assumptions. A high-bias model does not capture the true relationship well.

For example, suppose the relationship between age and cardiovascular risk is nonlinear, but we fit a logistic regression model with age only as a linear term. The model may underfit because it cannot represent the true shape of risk across age. Similarly, if treatment response depends on an interaction between tumour subtype and biomarker expression, a model that includes only main effects may miss an important pattern.

High-bias models tend to underfit. They may perform poorly both in training data and validation data.

7.2 Variance

Variance refers to error from excessive sensitivity to the training data. A high-variance model changes substantially when trained on a slightly different sample.

For example, a deep decision tree may create highly specific rules based on a few patients. It may perform very well on the training dataset but poorly on new patients. A random forest or gradient boosting model can also overfit if the dataset is small, the outcome is rare, or tuning is done incorrectly.

High-variance models tend to overfit. They may have excellent training performance but weak validation performance.

7.3 Irreducible error

Irreducible error is the part of the outcome that cannot be predicted from the available data. In medicine, irreducible error may arise from biological variability, unmeasured risk factors, measurement error, random events, incomplete clinical histories, or imperfect outcome definitions.

Even a perfectly developed model cannot predict everything. This is important for clinical communication. A prediction model estimates risk; it does not remove uncertainty.

8. Simple models and flexible models in medicine

A common misconception is that more complex models are always better. In medical machine learning, this is not true. More complex models may improve prediction in some settings, but they also increase the risk of overfitting, poor calibration, and weak interpretability.

8.1 Simple models

Simple models include linear regression, logistic regression, and Cox proportional hazards models. These models are familiar in medical statistics and often provide strong baselines. Their advantages include interpretability, stability, relatively transparent assumptions, and easier reporting.

A logistic regression model may perform surprisingly well for tabular clinical data. If the main relationships are approximately linear on the log-odds scale and the number of predictors is moderate, logistic regression may be hard to beat. It also produces predicted probabilities, which are central to clinical prediction.

8.2 Flexible models

Flexible models include decision trees, random forests, gradient boosting, support vector machines, and neural networks. These methods can capture nonlinearities and interactions without requiring all relationships to be specified manually. This can be useful when the data are complex and the sample size is sufficiently large.

However, flexible models are not automatically clinically superior. A model should be judged by validated performance, calibration, clinical usefulness, and applicability to the target setting. If a random forest gives only a tiny improvement in AUC over logistic regression but is poorly calibrated and harder to explain, it may not be preferable.

A practical principle is:

Use complexity only when it gives a meaningful and validated improvement for the clinical task.

9. Validation as a scientific requirement

Validation is the process of estimating how well a model performs outside the exact data used to develop it. In clinical prediction modelling, validation is part of the evidence that a model might be useful.

A model that has not been validated should be treated as a model hypothesis rather than a reliable clinical tool. It may suggest that prediction is possible, but it does not yet show that the model will work in practice.

Validation should answer several questions:

  • Does the model discriminate between patients with and without the outcome?
  • Are the predicted probabilities well calibrated?
  • Does performance remain acceptable when evaluated in new data?
  • Is performance consistent across clinically important subgroups?
  • Is the model useful at clinically relevant decision thresholds?
  • Does the model apply to the intended target population?

TRIPOD+AI emphasises transparent reporting of development and validation methods so that readers can judge the reliability and applicability of prediction model studies (Collins et al., 2024). PROBAST+AI further supports structured assessment of risk of bias and applicability (Moons et al., 2025).

10. Internal validation

Internal validation evaluates model performance using data from the same source as the development data. It estimates how much optimism may exist in the apparent performance and gives a better sense of how the model may perform in similar patients.

Common approaches include:

train/test split;

k-fold cross-validation;

repeated cross-validation;

bootstrap validation.

Internal validation is essential during model development. It helps avoid being misled by training performance. However, internal validation is not the same as external validation. It does not prove that the model will work in a different hospital, country, time period, or healthcare system.

11. Train/test split

A train/test split divides the available dataset into two parts. The training set is used to develop the model. The test set is held back and used only for evaluation.

A common split is:

Training set: 80%

Test set: 20%

For binary classification, the split should often be stratified so that the outcome prevalence is similar in both sets. For example, if 10% of patients have the outcome, a stratified split tries to preserve that 10% proportion in both training and test data.

A train/test split is easy to understand, but it has limitations. If the dataset is small, the test set may contain too few outcome events to estimate performance precisely. The result can depend on the random split. For small or moderate datasets, cross-validation or bootstrap validation may provide more efficient internal validation.

12. Cross-validation

Cross-validation divides the data into several folds. In 5-fold cross-validation, the data are split into five parts. The model is trained on four parts and tested on the remaining part. This is repeated five times so that each part is used once as a validation fold. The performance estimates are then averaged.

Cross-validation is useful because it allows each observation to be used for both training and validation at different stages. It is especially helpful when data are limited.

However, cross-validation must be done carefully. All steps that learn from data must occur inside the cross-validation loop. This includes imputation, scaling, feature selection, PCA, model tuning, and threshold selection. If these steps are performed before cross-validation using the full dataset, information from the validation folds leaks into training.

This is one of the most common mistakes in biomedical machine learning.

13. Bootstrap validation

Bootstrap validation is often used in clinical prediction modelling. It involves repeatedly sampling from the development dataset with replacement, fitting the model in the bootstrap sample, and evaluating optimism by comparing performance in the bootstrap sample and original data.

Bootstrap methods can be particularly useful when the dataset is not large enough to set aside a separate test set. Steyerberg (2019) discusses bootstrap validation as a practical method for estimating and correcting optimism in prediction model performance.

The key idea is that bootstrap validation tries to estimate how much the model’s apparent performance is inflated by overfitting. This optimism-corrected estimate is usually more realistic than apparent performance.

14. External validation

External validation evaluates a model using independent data from a different source. This is one of the strongest tests of a prediction model.

Examples include:

developing a model in Hospital A and testing it in Hospital B;

developing a model in one country and testing it in another;

developing a model using data from 2015–2020 and testing it in 2021–2024;

developing a model in a clinical trial and testing it in real-world registry data;

developing a model in one cohort and testing it in another cohort.

External validation is important because medical data are highly context-dependent. Disease prevalence, treatment pathways, measurement systems, coding practices, population demographics, and healthcare access may all differ across settings. A model that performs well in one dataset may perform poorly elsewhere.

A model without external validation may still be valuable as a development-stage model, but it should not be presented as broadly generalisable.

15. Generalisation and transportability

Generalisation refers to whether a model performs well in new individuals from the same broad population. Transportability refers to whether a model performs well in a different but related population or setting.

For example, a model developed using data from one NHS hospital may generalise reasonably to similar NHS hospitals, but may not transport well to another country with different patient demographics, clinical pathways, laboratory systems, and outcome coding. A model developed in adults may not transport to children. A model developed in trial participants may not transport to routine clinical populations, because trial participants are often more selected.

This is why the target population must be defined clearly. A prediction model is not simply “good” or “bad” in isolation. It is good or bad for a specific prediction task, at a specific prediction time, in a specific population, using specific predictors.

16. Calibration and validation

Validation should not focus only on discrimination. A model may have a good AUC but poorly estimated probabilities. In clinical medicine, predicted probabilities often matter because decisions are based on absolute risk.

Calibration measures agreement between predicted risks and observed risks. For example, among patients predicted to have a 20% risk, approximately 20% should experience the outcome if the model is well calibrated in that risk range.

Poor calibration can be harmful. If a model overestimates risk, patients may receive unnecessary monitoring, anxiety, or treatment. If it underestimates risk, high-risk patients may be falsely reassured or denied preventive care. Steyerberg (2019) emphasises that prediction models should be assessed using both discrimination and calibration.

When validating a model, we should therefore ask:

  • Does the model rank patients correctly?
  • Are predicted probabilities accurate?
  • Does calibration vary across subgroups?
  • Does calibration remain acceptable in external data?

Would recalibration be needed before clinical use?

17. Data leakage: the hidden threat

Data leakage occurs when information is used during model development that would not be available in real prediction, or when information from validation/test data is allowed to influence model training.

Leakage is dangerous because it can produce unrealistically high performance. A model affected by leakage may appear excellent in a paper, but fail when used in real clinical practice.

Leakage can be obvious, such as including discharge destination when predicting mortality at admission. It can also be subtle, such as selecting features using the full dataset before cross-validation.

A good practical question is:

Would this information be available, in this form, at the moment the prediction is supposed to be made?

If the answer is no, the variable may cause leakage.

18. Leakage from future information

Future information leakage occurs when predictors include information measured after the prediction time.

Suppose the clinical question is:

Can we predict ICU mortality at the time of ICU admission?

Appropriate predictors might include age, sex, admission diagnosis, blood pressure at admission, oxygen saturation at admission, baseline laboratory values, and comorbidity history.

Leakage-risk predictors would include ICU length of stay, discharge destination, final diagnosis after discharge, treatment response after several days, laboratory values measured shortly before death, and interventions performed after deterioration.

These variables may be strongly associated with mortality, but they are not available at ICU admission. Including them would make the model unrealistic and inflate performance.

19. Leakage from preprocessing

Preprocessing leakage occurs when steps such as scaling, imputation, feature selection, or PCA are performed using the full dataset before splitting or cross-validation.

For example, suppose a researcher standardises all predictors using the mean and standard deviation from the full dataset, and then performs train/test splitting. The test set has already influenced the preprocessing parameters. This is leakage.

The correct approach is:

Split the data into training and test sets.

Estimate preprocessing parameters using the training set only.

Apply those parameters to the test set.

In cross-validation, the same principle applies within each fold. Preprocessing must be learned from the training fold and applied to the validation fold.

This is why workflow tools are important. In R, tidymodels recipes and workflows help keep preprocessing inside cross-validation, reducing the risk of data leakage.

20. Leakage from feature selection

Feature selection leakage is one of the most serious problems in omics and high-dimensional biomedical machine learning.

Consider a dataset with:

n = 250 participants p = 20,000 gene expression features

An incorrect workflow would be:

Use the full dataset to select the 100 genes most associated with disease status.

Perform cross-validation using only those 100 selected genes.

Report the cross-validated AUC.

This workflow is biased because the validation folds contributed to feature selection. The selected features were chosen partly because of their relationship with outcomes in the validation data. Therefore, the cross-validation result is no longer an honest estimate of performance.

The correct workflow is:

Split the data into cross-validation folds.

Within each training fold, perform feature selection using only that training fold.

Fit the model using the selected features from that training fold.

Evaluate on the held-out validation fold.

Repeat for each fold.

For a final model, feature selection should be repeated using the training data only, and performance should be evaluated on an untouched test set or external dataset.

21. Leakage from duplicate or related observations

Medical datasets often contain repeated or related observations. A patient may have multiple admissions, multiple clinic visits, multiple images, multiple tissue samples, or repeated biomarker measurements. If rows are split randomly, the same patient may appear in both training and test sets.

This creates leakage because the model may recognise patient-specific patterns rather than general disease patterns.

For example, if a dataset contains multiple images per patient and images are randomly split, images from the same patient may appear in both training and test sets. The model may perform well because it has effectively seen the patient before.

The correct approach is to split by patient, not by row. In clustered data, splitting may need to account for hospital, family, batch, centre, or other grouping structures.

22. Leakage from outcome proxies

Some variables are not the outcome itself but are very close proxies for it. These variables may create unrealistic performance if they would not be available in real prediction or if they encode clinical knowledge that occurs after the prediction point.

Examples include:

palliative care referral when predicting short-term mortality;

discharge code when predicting diagnosis at admission;

treatment escalation when predicting deterioration;

“number of days in ICU” when predicting ICU mortality;

clinician risk category assigned after reviewing the full case;

follow-up imaging results when predicting baseline prognosis.

These variables may seem legitimate because they are present in the dataset. But prediction modelling is not about using every available column. It is about using predictors that would be available and appropriate at the intended prediction time.

23. Leakage in survival prediction

Survival prediction has additional leakage risks because time is central. Suppose we want to predict 5-year survival at cancer diagnosis. Predictors should be available at or near diagnosis.

Appropriate predictors may include age at diagnosis, tumour stage, tumour grade, receptor status, baseline comorbidities, and baseline biomarkers.

Potential leakage variables include recurrence during follow-up, treatment response after chemotherapy, follow-up imaging, later metastasis status, end-of-life care variables, and survival time itself.

A survival model should respect the temporal ordering of information. If a variable is measured after the prediction time, it should not be used for baseline prediction.

24. Leakage in EWAS and omics prediction

In EWAS, gene expression, methylation, proteomics, and metabolomics studies, leakage can be particularly subtle. The data are high-dimensional, preprocessing is complex, and batch effects are common.

Potential leakage sources include:

selecting CpG sites using the full dataset before validation;

performing PCA using all samples before splitting;

carrying out batch correction using outcome information;

normalising using the full dataset before cross-validation;

allowing duplicate samples from the same participant in different folds;

using technical processing variables that are confounded with outcome status;

tuning the model repeatedly using the test set.

A careful omics machine learning workflow should define the split first, carry out preprocessing within resampling, perform feature selection inside the training folds, and reserve an untouched test set or external dataset for final evaluation. This is especially important when p >> n.

25. Warning signs in medical machine learning papers

When reading a medical machine learning paper, be cautious if you see any of the following:

extremely high accuracy from a small dataset;

performance reported only on training data;

no internal validation;

no external validation;

feature selection before splitting;

unclear prediction time point;

predictors measured after the outcome;

no calibration assessment;

only AUC reported;

no missing data strategy;

no model specification;

no description of hyperparameter tuning;

no code or reproducibility information;

no assessment of applicability to the target population.

These issues do not automatically mean the study is useless, but they reduce confidence in the reported performance. PROBAST+AI is designed to help evaluate these kinds of bias and applicability concerns systematically (Moons et al., 2025).

26. Practical checklist before fitting a medical ML model

Before fitting a model, ask:

  • What is the clinical prediction question?
  • What is the target population?
  • What is the exact prediction time point?
  • What is the outcome and how is it measured?
  • Are all predictors available at prediction time?
  • Are there predictors that encode future information?
  • Are there repeated observations per patient?
  • Is the outcome rare?
  • Is the sample size adequate for the number of predictors?
  • Will preprocessing be performed inside resampling?
  • Will feature selection be performed inside resampling?
  • How will discrimination be assessed?
  • How will calibration be assessed?
  • Is external validation possible?
  • How will risk of bias and applicability be discussed?
  • How will the model be reported transparently?

This checklist should become a habit. It prevents many of the most common mistakes in medical machine learning.

27. Worked example: hospital readmission prediction

Suppose we want to predict 30-day hospital readmission at the time of discharge.

The prediction time is discharge. Therefore, predictors available up to discharge may be allowed, but predictors after discharge should not be used.

Appropriate predictors may include age, sex, admission diagnosis, comorbidities, length of stay, discharge medications, laboratory values before discharge, previous admissions, and discharge destination if it is genuinely known at discharge and part of the decision context.

Leakage-risk predictors may include readmission scheduling information, follow-up events after discharge, post-discharge emergency visits, or variables updated after the readmission occurs.

Validation should ideally assess whether the model works in later time periods or another hospital. Calibration is important because the model may be used to identify patients who need additional support. A model that overestimates readmission risk may waste resources; a model that underestimates risk may miss patients who would benefit from intervention.

28. Worked example: baseline cancer prognosis

Suppose the aim is to predict 5-year survival for patients newly diagnosed with breast cancer.

The prediction time is diagnosis or baseline assessment. Predictors should be measured at or before that time. These might include age, tumour size, grade, stage, receptor status, baseline comorbidities, and baseline biomarkers.

If the dataset contains variables such as recurrence during follow-up, response to chemotherapy, later metastasis, or palliative care referral, these should not be used for baseline prediction. They occur after the prediction time and would create leakage.

A useful validation strategy might include internal validation using bootstrap or cross-validation, followed by external validation in another cancer registry or cohort. Performance should include discrimination, calibration at relevant time points, and possibly decision curve analysis if the model is intended to guide treatment or follow-up intensity.

29. Worked example: methylation-based disease classification

Suppose the aim is to classify disease status using DNA methylation data. The dataset contains 300 participants and 450,000 CpG sites.

This is a high-dimensional problem. The risk of overfitting is high because there are far more predictors than observations. A naive analysis may identify CpG sites that appear strongly associated with disease purely by chance.

A poor workflow would select CpG sites using the full dataset, then cross-validate a classifier. This leaks outcome information into the validation folds.

A better workflow would define cross-validation folds first, perform feature filtering and model fitting within each training fold, and evaluate on the held-out fold. If a final test set is available, it should remain untouched until the end. External validation in another cohort would be even stronger.

The biological interpretation should also be cautious. A feature selected by a prediction model is not automatically causal. It may be a proxy, a batch-related artefact, or unstable across cohorts.

30. Summary

Validation, overfitting, and leakage are core concepts in machine learning for biostatistics. They determine whether reported model performance is believable.

A model can look excellent in the development dataset but fail in new patients. This may happen because of overfitting, high variance, leakage, population differences, measurement differences, or poor calibration.

The key principles are:

Training performance is not enough.

Apparent performance is usually optimistic.

Internal validation is essential during development.

External validation is important for clinical generalisability.

Preprocessing and feature selection must occur inside resampling.

Predictors must be available at the intended prediction time.

Calibration must be assessed, not only discrimination.

Risk of bias and applicability should be considered from the start.

In medical machine learning, validation is not a technical afterthought. It is part of responsible scientific practice.

Key takeaway

A medical machine learning model is not credible simply because it performs well on the development dataset. It becomes credible only when it is developed without leakage, validated appropriately, assessed for calibration and clinical usefulness, and reported transparently enough for others to judge its risk of bias and applicability.

R practical: training performance, test performance and leakage

This script introduces a simple train/test split using the Pima Indians Diabetes dataset. It fits a logistic regression model, compares training and test performance, then demonstrates how a leakage variable can make performance look unrealistically strong.

What the script teaches

Training AUC, test AUC, classification metrics and why test performance matters.

Leakage demonstration

Clinical interpretation

# Machine Learning in Biostatistics
# Script 02: Validation, Overfitting and Data Leakage
#
# This script introduces the idea that training performance can look better
# than test performance.
#
# We use the Pima Indians Diabetes dataset because it is simple, familiar,
# and useful for teaching binary clinical prediction.
#
# Main ideas:
# - split data into training and test sets
# - fit a logistic regression model
# - compare training and test performance
# - demonstrate why test performance matters
# - show how leakage can make performance look artificially strong

library(tidyverse)
library(mlbench)
library(pROC)

set.seed(123)

data("PimaIndiansDiabetes")

diabetes_data <- PimaIndiansDiabetes %>%
  mutate(
    diabetes = factor(diabetes, levels = c("neg", "pos"))
  )

# Quick look at the data

glimpse(diabetes_data)

diabetes_data %>%
  count(diabetes)

# The outcome is diabetes status:
# neg = diabetes negative
# pos = diabetes positive
#
# The positive class is "pos".

# Train/test split

n <- nrow(diabetes_data)

train_index <- sample(
  x = seq_len(n),
  size = round(0.80 * n)
)

train_data <- diabetes_data[train_index, ]
test_data <- diabetes_data[-train_index, ]

train_data %>% count(diabetes)
test_data %>% count(diabetes)

# Fit a simple logistic regression model

model_basic <- glm(
  diabetes ~ pregnant + glucose + pressure + triceps +
    insulin + mass + pedigree + age,
  data = train_data,
  family = binomial
)

summary(model_basic)

# Predicted probabilities in training data

train_data <- train_data %>%
  mutate(
    predicted_risk = predict(
      model_basic,
      newdata = train_data,
      type = "response"
    )
  )

# Predicted probabilities in test data

test_data <- test_data %>%
  mutate(
    predicted_risk = predict(
      model_basic,
      newdata = test_data,
      type = "response"
    )
  )

# AUC in training data

train_roc <- roc(
  response = train_data$diabetes,
  predictor = train_data$predicted_risk,
  levels = c("neg", "pos")
)

train_auc <- auc(train_roc)

# AUC in test data

test_roc <- roc(
  response = test_data$diabetes,
  predictor = test_data$predicted_risk,
  levels = c("neg", "pos")
)

test_auc <- auc(test_roc)

performance_comparison <- tibble(
  dataset = c("Training data", "Test data"),
  auc = c(as.numeric(train_auc), as.numeric(test_auc))
)

print(performance_comparison)

# Interpretation:
#
# Training performance is usually optimistic because the model was fitted
# using the training data.
#
# Test performance is more useful because the test data were not used to
# fit the model.
#
# In medical prediction, we care about performance on future patients,
# not only on the patients used for model development.

# Convert probabilities to predicted classes using a 0.50 threshold

threshold <- 0.50

train_data <- train_data %>%
  mutate(
    predicted_class = ifelse(predicted_risk >= threshold, "pos", "neg"),
    predicted_class = factor(predicted_class, levels = c("neg", "pos"))
  )

test_data <- test_data %>%
  mutate(
    predicted_class = ifelse(predicted_risk >= threshold, "pos", "neg"),
    predicted_class = factor(predicted_class, levels = c("neg", "pos"))
  )

# Simple function to calculate classification metrics

calculate_metrics <- function(actual, predicted) {
  tab <- table(predicted, actual)

  tn <- tab["neg", "neg"]
  fp <- tab["pos", "neg"]
  fn <- tab["neg", "pos"]
  tp <- tab["pos", "pos"]

  accuracy <- (tp + tn) / sum(tab)
  sensitivity <- tp / (tp + fn)
  specificity <- tn / (tn + fp)

  tibble(
    accuracy = accuracy,
    sensitivity = sensitivity,
    specificity = specificity
  )
}

train_metrics <- calculate_metrics(
  actual = train_data$diabetes,
  predicted = train_data$predicted_class
)

test_metrics <- calculate_metrics(
  actual = test_data$diabetes,
  predicted = test_data$predicted_class
)

metric_comparison <- bind_rows(
  train_metrics %>% mutate(dataset = "Training data"),
  test_metrics %>% mutate(dataset = "Test data")
) %>%
  select(dataset, accuracy, sensitivity, specificity)

print(metric_comparison)

# Demonstrating a leakage problem

# Now we create an artificial leakage variable.
# This variable is almost the outcome itself, but with slight noise.
#
# In real medical data, leakage may be less obvious.
# For example, a post-diagnosis variable might accidentally be used
# to predict diagnosis at baseline.

diabetes_data_leakage <- diabetes_data %>%
  mutate(
    leakage_variable = ifelse(diabetes == "pos", 1, 0),
    leakage_variable = leakage_variable + rnorm(n(), mean = 0, sd = 0.05)
  )

train_data_leakage <- diabetes_data_leakage[train_index, ]
test_data_leakage <- diabetes_data_leakage[-train_index, ]

model_with_leakage <- glm(
  diabetes ~ pregnant + glucose + pressure + triceps +
    insulin + mass + pedigree + age + leakage_variable,
  data = train_data_leakage,
  family = binomial
)

test_data_leakage <- test_data_leakage %>%
  mutate(
    predicted_risk = predict(
      model_with_leakage,
      newdata = test_data_leakage,
      type = "response"
    )
  )

leakage_roc <- roc(
  response = test_data_leakage$diabetes,
  predictor = test_data_leakage$predicted_risk,
  levels = c("neg", "pos")
)

leakage_auc <- auc(leakage_roc)

leakage_comparison <- tibble(
  model = c("Correct model", "Model with leakage"),
  test_auc = c(as.numeric(test_auc), as.numeric(leakage_auc))
)

print(leakage_comparison)

# Interpretation:
#
# The leakage model will look extremely strong because it contains a variable
# that directly encodes the outcome.
#
# This is not a useful model.
#
# In real clinical prediction, leakage can occur when we use information
# that would not be available at the prediction time.
#
# Examples:
# - using discharge information to predict admission risk
# - using follow-up test results to predict baseline diagnosis
# - selecting features before validation
# - preprocessing the full dataset before train/test split

# Plot ROC curves for correct model and leakage model

plot(
  test_roc,
  col = "black",
  lwd = 2,
  main = "ROC Curve: Correct Model vs Leakage Model"
)

plot(
  leakage_roc,
  col = "gray40",
  lwd = 2,
  add = TRUE
)

abline(a = 0, b = 1, lty = 2)

legend(
  "bottomright",
  legend = c(
    paste0("Correct model, AUC = ", round(test_auc, 3)),
    paste0("Leakage model, AUC = ", round(leakage_auc, 3))
  ),
  lwd = 2,
  col = c("black", "gray40"),
  bty = "n"
)

# Final learning points

cat("\nLearning points:\n")
cat("1. Training performance is usually optimistic.\n")
cat("2. Test performance is more useful for estimating future performance.\n")
cat("3. Data leakage can make a model look unrealistically strong.\n")
cat("4. Predictors must be available at the intended prediction time.\n")
cat("5. Validation is central to medical machine learning.\n")

Exercise 6: Leakage from preprocessing

A researcher standardises predictors using the full dataset, then splits into training and test sets.

Click to see answer

The workflow is wrong because the test data influenced the means and standard deviations used for standardisation.

The correct workflow is: split first, estimate preprocessing parameters on the training data only, then apply those same parameters to the test data. In cross-validation, this must happen separately inside each training fold.

Exercise 7: Leakage from feature selection

A researcher selects 100 genes using the full dataset, then performs 10-fold cross-validation using only those genes.

Click to see answer

This is problematic because the validation folds influenced feature selection. The cross-validation estimate is therefore optimistic.

Feature selection should be performed inside each training fold only. The held-out validation fold should not influence which genes are selected.

Exercise 10: Short answer questions

Why is apparent performance usually optimistic?

Click to see answer

Apparent performance is measured in the same data used to develop the model. It includes the benefit of model fitting, tuning, variable selection and other development decisions, so it is usually higher than performance in new data.

Why is external validation important?

Click to see answer

External validation tests whether the model works in a different setting, time period or population. This is essential because clinical data and practice can vary across hospitals and populations.

Why can overfitting be severe in omics data?

Click to see answer

Omics datasets often have many more predictors than observations. This makes it easy to find chance patterns that look predictive in the development data but do not generalise.

Why should preprocessing be performed inside cross-validation?

Click to see answer

Preprocessing can learn information from data. If it is done before cross-validation, the validation folds influence the training process. Performing preprocessing inside each training fold avoids leakage.

Why is it dangerous to include predictors measured after the prediction time?

Click to see answer

Such predictors contain future information. They may make the model look strong during evaluation, but they would not be available when the model is used in real practice.

Exercise 11: Mini case study

A team develops a model to predict 1-year mortality after cancer diagnosis.

  • Sample size: 150 patients
  • Predictors: 2,000 biomarkers
  • Model: random forest
  • Training AUC: 0.99
  • No test set
  • No cross-validation
  • No calibration plot
  • No external validation
Click to see answer

Concerns: small sample size, high-dimensional predictors, flexible model, training-only AUC, no internal validation, no external validation, no calibration assessment, high risk of overfitting, unclear preprocessing and feature-selection workflow.

Recommended validation: internal validation using nested cross-validation or bootstrap validation, with all preprocessing and feature selection inside resampling. If possible, reserve an untouched test set or validate externally in another cohort.

Additional reporting: prediction time point, outcome definition, predictor availability, missing data handling, tuning process, discrimination, calibration, clinical usefulness, risk of bias, applicability and reproducible code.
